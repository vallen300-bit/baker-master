0System in one paragraph

Baker is a multi-agent institution. Sentinel layer: Baker (always-on FastAPI service) ingests email/WhatsApp/meetings/ClickUp/RSS, embeds to Qdrant, writes raw tables, serves memory via the Baker MCP (~47 tools). Reasoning+orchestration layer: a fleet of Claude Code agents + Codex (cross-vendor verifier) coordinating over the Brisen Lab bus. Authority layer: Director is final authority and talks to AH1; AH1 orchestrates under A/B/C tiers. Reasoning target is Cortex 3T (per-matter 6-phase cycles), migration in flight. LLM: claude-opus-4-8.

4Delivery lifecycle (9 stages)

  1. Brief — AH1 authors via write-brief skill (problem → constraints → AC).
  2. Dispatch — mailbox CODE_<N>_PENDING.md + same-turn bus-post.
  3. Claim — B-code orients (brief + memory), confirms on bus.
  4. Implementation — surgical, tests-first, fault-tolerant.
  5. Ship report — literal pytest output; no "by inspection".
  6. Review gates — G0 Codex pre-review (cross-vendor) → G1 AH1 fold / gate-1 static review → G2 /security-review / gate-2 where required → G3 AH2 deputy plus deeper gate-3 / gate-4 reviews where applicable.
  7. Merge — AH1 squash-merge on green gates.
  8. Deploy — Render auto-deploys from main.
  9. Post-deploy AC + writeback — verify ACs live → PINNED update, activity log, cascade-back-prop, lessons.

Gate naming: v6 uses G0-G3 for the lifecycle. Existing ops skills still use gate-1 through gate-4. Map G1 to gate-1 static review, G2 to gate-2 /security-review, and G3 to deputy / architecture / code-reviewer deeper gates where applicable.

Security-review precision: /security-review is mandatory for Tier-B PRs and executable, DB, security, auth, API, webhook, external-surface, email, WhatsApp, or Render-touching changes. Docs-only / architecture-only work may skip it when the diff is prose or static HTML only and the skip reason is recorded.

5Harness V2 implications

  1. The bus is the load-bearing primitive, not the MCP. Model it as the spine — dispatch, gates, heartbeats, acks all ride it.
  2. Roles are tiered, not flat. Encode A/B/C + Director-facing-or-not as first-class. B-codes/Codex structurally cannot address Director.
  3. Review is multi-vendor, multi-gate. G0–G3 is the default path = the correctness immune system.
  4. Memory is two-layer: Baker MCP (live operational) vs vault curated (per-matter, persistent). Do not conflate.
  5. Cortex 3T is the reasoning target: per-matter cycles, not one global thinking session.
  6. Cheap/disposable build layer (B-codes, Researcher fan out); singular/stateful orchestration (AH1).

6Hard invariants (agents respect these)

Bus-post on every state change — ship, dispatch, blocker, heartbeat, gate verdict, merge, Tier-B ask/done. Use dispatched_by/reply_to; do not route every reply blindly to lead.
No "by inspection" — ship on literal green test run only.
Tier boundaries are hard — never exceed your tier even if confident; B-codes never Director-facing.
Fault-tolerant or it doesn't ship — all DB/API calls wrapped; fail loud, never skip silently.
