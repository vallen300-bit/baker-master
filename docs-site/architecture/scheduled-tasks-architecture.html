Architecture Review — Director Draft

Scheduled Tasks for Agent Fleet

Problem · Lifecycle · Options · Recommendation · Engineering handoff
Prepared by MOVIE Desk
12 May 2026
Director review → AH1 engineering eval
Bottom line
We do not need a third-party task manager. The agent fleet needs a structured, queryable, agent-readable store for tasks and scheduled jobs — and we already have it (the baker-vault, plus Baker's existing deadline system). Three storage tiers, one messenger: (1) Obsidian-vault markdown as canonical source of truth for soft per-desk tasks; (2) Baker deadline system (baker_add_deadline) for hard counterparty deadlines that must nudge cross-session; (3) Baker APScheduler for recurring agent jobs. All three surface to the Director via the single Baker → Slack DM channel. Drop ClickUp. Keep Google Calendar for time-blocked meetings only. Director never opens Obsidian / Todoist / ClickUp for pings — Baker is the only messenger.
Section 1 · The problem

What we are trying to solve, and what is breaking

What the agent fleet needs

Across the desk fleet (MOVIE Desk, AO Desk, Hagenauer Desk, Baden-Baden Desk, AI Head, AI Dennis, future desks), there are three distinct kinds of scheduled item — each with different storage and surfacing requirements. Today none is consistently met.

  • Soft per-desk tasks — "tomorrow morning, debrief Director on the MOHG meeting"; "Friday, send draft to Mario"; "this week, review the Sonderwunsch list." Per-desk, per-matter, agent-owned. Read at session start; closed when the agent acts. These need a place that lives next to the matter context.
  • Hard counterparty deadlines — "Skliar response due 30.06.2026"; "Aukera covenant test 30.09.2026"; "Eastdil HMA extract owed 16 May"; "residence fee deferral year-end 31.12.2026." Externally fixed dates with counterparty consequences. Must nudge cross-session, regardless of which desk happens to be open. Need a system-wide store that surfaces to Director independently of any one Desk being active. This category surfaced via Baden-Baden Desk proposal 12 May; previously missing from the architecture.
  • Recurring agent jobs — "every Monday 09:00, MOVIE Desk produces weekly status"; "monthly, reconcile residence-fee ledger"; "weekly, sweep counterparty inbox for stale threads." Cron-like triggers that fire an agent on a schedule.

What is breaking today

  • ClickUp via MCP is one-way. Create, update, move — yes. Delete folder or list — no tool exposed. Confirmed today during MOVIE Desk setup. Long-term this turns the Director into a manual janitor for AI-created clutter. Unacceptable.
  • Notion via MCP is broken. Memory creates duplicates instead of editing (recorded in Baker MEMORY.md). Off the list.
  • Todoist via MCP works but is personal-task-shaped. Full CRUD via API v1. Single-project hierarchy. No native concept of agents, matters, or curated context. Strong mobile UX — which the Director has explicitly said he does not need to attend to.
  • Google Calendar via MCP works but is time-blocked-event-shaped. Wrong tool for "by Friday" tasks; right tool for "10:00 Tuesday meeting."
  • ClickUp / Todoist / Notion all introduce a third-party hop between the agent's filesystem (where context lives) and the task store. Every hop is a place where MCP gaps appear (delete, query, update, sync).

The underlying issue. We have been trying to pick a SaaS task manager and bolt the agent fleet onto it. The agent fleet's centre of gravity is the baker-vault (markdown, git-tracked, AI has direct filesystem control). Every time we route tasks through a SaaS tool, we lose agent control at the boundary. The right architecture flips this: the vault is the source of truth, the SaaS tools (if any) are downstream surfaces.

Section 2 · Lifecycle of a scheduled task

The five steps every task goes through

Any architecture must support each step cleanly. Most tools are strong at 2–3 steps and weak at the rest — which is what creates seams in practice.

Creation

An agent (or the Director, by speaking to an agent) declares a task: title, due date or recurrence, owner agent, expected output, references to context.
Where the task is born determines who can act on it later. If the agent creates it but cannot read it back, the system fails.

Storage

The task lives somewhere queryable. Ideally with the context it relates to — emails, decisions, matter state — not in a separate silo.
Tasks decay value when separated from their context. A task that reads "debrief MOHG meeting" is worth ten times more when it sits next to the meeting agenda, the talking points, and the prior decisions log.

Reading at the right time

A scheduler or agent picks up the task when it is due — at session start, on cron trigger, on autonomous loop, or on inbound event (new email, ratification).
The Director's principle: he does not attend to the schedule. The agent reads it. The system must therefore support agent-side queries: "what is due today?", "what is overdue?", "what is mine?"

Execution

The owner agent runs the task: drafts a brief, sends a paste-block, produces a debrief, files a ratification request. Output lands in the vault and is surfaced to the Director via the agreed channel.
Execution must be auditable. Every task touches at least one file write, one decision-log entry, one ratification record.

Closure and audit trail

The agent marks the task complete (or cancelled, or deferred), with a one-line outcome and a link to what was produced. The closed task remains in the vault as history.
Without closure, the queue grows. Without an audit trail, the Director cannot challenge or trust the system. Both must work via filesystem operations the agent fully controls.
Section 3 · Options evaluated

What is on the table

Eight candidates evaluated against the five-step lifecycle, plus the Director's stated frame (Baker is the messenger; agents read the schedule; Director receives a single curated Slack stream).

A · Obsidian vault markdown (filesystem) Recommend
Tasks as markdown files in baker-vault/_ops/agents/<desk>/tasks/, with frontmatter (due, priority, owner, status, references). Agents read, write, update, delete directly via filesystem. Director views in Obsidian.
Pros
  • Full agent control. Read, write, edit, delete — no MCP gap. Filesystem is the API.
  • Tasks sit next to the matter context they reference. Click a link, get everything.
  • Git-tracked. Every state change auditable.
  • DataView / Tasks plugins give queryable dashboards inside Obsidian.
  • Already the Baker substrate. No new system to maintain.
  • No third-party dependency. No vendor risk. No MCP version risk.
  • Director-facing reading surface (Obsidian app on Mac) is already in place — used by Director on demand, not required for pings.
  • Notification responsibility lives with Baker, not Obsidian. Baker reads the vault and pings the Director via Slack. Obsidian is the data store; it is never the messenger.
Cons
  • Tasks plugin / DataView add useful querying but introduce small plugin-maintenance overhead.
  • If many agents write concurrently, baker-vault shared-FS race conditions are a known historical issue (mitigated by branch-per-agent worktrees).
Fit: primary store for matter-anchored tasks and follow-ups. Lifecycle steps 1–5 all native. Director's stated frame ("I don't want to be pinged by Obsidian / Todoist / ClickUp — I can be pinged by Baker") fits this option uniquely well, because Obsidian is data not messenger and Baker is the single Director-facing channel.
B · Baker deadline system (baker_add_deadline) Recommend (paired with A)
Existing Baker primitive for hard, date-fixed deadlines. Tools: mcp__baker__baker_add_deadline to register, baker_deadlines to query. State in Postgres. Surfaces cross-session via Baker's existing nudge logic (Slack DM at the right cadence ahead of the deadline). Surfaced by Baden-Baden Desk 12 May as the right home for counterparty deadlines that must fire regardless of which desk is in session.
Pros
  • Cross-session nudges. Skliar 30.06.2026 fires even if Baden-Baden Desk hasn't been opened for weeks. The deadline owns its own surfacing logic — does not depend on a desk being active.
  • Already in production. No new infrastructure.
  • Full CRUD via Baker MCP. Add, query, dismiss, close.
  • Logged to Baker's audit trail by default.
  • Pairs cleanly with vault tasks: the deadline is the trigger; the vault is the context.
Cons
  • Best for fixed dates, not "by Friday-ish" tasks (those belong in the vault).
  • Need an explicit rule for what counts as "hard deadline" — otherwise the channel pollutes with soft items.
  • Existing surfacing cadence (lead-time before deadline, escalation pattern) needs confirmation — may need refinement for desk-fleet use.
Fit: the right home for hard counterparty deadlines (court dates, contract expiries, regulatory deadlines, covenant tests, statutory cut-offs). Use rule: "if missing this date has a counterparty consequence, it goes here. If it's an internal action that can flex by days, it goes in the vault."
C · Baker-native scheduler (APScheduler + Postgres) Recommend (paired with A + B)
Recurring jobs and time-based triggers run via APScheduler on Render. Job definitions live in Baker code (or a YAML registry); state in Postgres. Already in production for AI Head weekly audit, Sunday digest, Todoist polling, cache TTL.
Pros
  • Already running. No new infrastructure.
  • Cron-precise triggers (e.g. Monday 09:00 UTC).
  • Jobs fire agent runs directly — no human in the loop.
  • Tier B authority charter already governs what APScheduler jobs are allowed to do.
  • Fully auditable via baker_actions rows.
Cons
  • Job registry is engineer-edited (YAML or code) — not a Director surface. AH1 / B-codes own changes.
  • No Director-facing view of "what's scheduled." Mitigated by a small auto-generated digest.
  • Requires the agent it fires to actually be runnable on a Render trigger (today this means Codex agents, not Claude desktop sessions).
Fit: right tool for recurring agent jobs and time-based triggers. Pairs with Option A: APScheduler reads Obsidian tasks at the scheduled time and fires the right desk. Wraps lifecycle step 3 (reading at the right time) for recurring patterns.
D · Google Calendar (via MCP) Keep — narrow scope
Native calendar events. Full CRUD via MCP. Already integrated. Already where Director has time-blocked meetings.
Pros
  • Full CRUD via MCP (create, update, move, delete).
  • Director sees it natively in the same calendar he already uses.
  • Right tool for time-blocked items (meetings, calls, on-site visits).
  • Supports recurrence, reminders, invitees.
Cons
  • Wrong tool for non-time-blocked tasks ("by Friday", "after Q2 results"). Not queryable as a task list.
  • Calendar is a Director-attended surface. Agents writing to it pollutes Director's day view.
  • No matter-context anchoring.
Fit: meetings and time-blocked items only. Agent draft a meeting → Director ratifies → agent creates event. Do not use as a general task store.
E · Todoist (via MCP) Viable for personal-life tasks only
SaaS task manager with full CRUD via API v1. Already MCP-integrated. Baker indexes every 30 minutes. Director uses it for personal tasks today.
Pros
  • Full CRUD via MCP. No gaps.
  • Excellent mobile UX, native push, widgets.
  • Baker already polling and indexing.
Cons
  • Director-attended by design. The whole UX is built for the human user to read and tick. The Director's frame is the opposite — he does not want to attend to the schedule.
  • Flat project hierarchy. No matter-context anchoring.
  • Single-user. Agents cannot easily own tasks the way they can in the vault.
  • Adds a third-party hop between agent filesystem and task store.
Fit: keep for the Director's personal-life tasks. Do not use for desk fleet work.
F · ClickUp (via MCP) Drop
Project management SaaS. Full CRUD on tasks via MCP. Folders and lists are create-only — no delete tool exposed.
Pros
  • Hierarchical structure (space → folder → list → task) fits multi-desk taxonomy in principle.
  • Tasks have rich custom fields, priorities, statuses.
  • Already populated with legacy Brisen data.
Cons
  • Delete gap. No MCP tool to remove folders or lists. Confirmed 12 May 2026. Every AI-created mistake becomes Director's manual janitor job.
  • No filesystem-level write. Agents must reach through MCP for every operation.
  • Director must learn ClickUp UI to do cleanup — not a strategy-time activity.
  • Existing Brisen ClickUp content is legacy and partly stale; cleaning it is a separate project.
Fit: drop as scheduling infrastructure. Keep existing legacy data in place (no migration); stop accumulating new desk work there.
G · Notion (via MCP) Reject — broken
Document and database SaaS. MCP integration exists but has a known bug.
Pros
  • Strong document + database hybrid model.
Cons
  • Update operation creates duplicates instead of editing. Recorded in Baker MEMORY.md.
  • Any agent that writes will pollute the workspace.
  • Vendor MCP fix timeline unknown.
Fit: off the table until the duplicate bug is fixed upstream.
H · Linear (via MCP) Wrong tool
Engineering issue tracker. Full CRUD via API. MCP available.
Pros
  • Full CRUD.
  • Strong workflow states, projects, cycles.
  • Good API.
Cons
  • Designed for software engineering teams. The vocabulary (issues, epics, cycles, projects) does not map to asset-management work.
  • Adds a third-party hop with no matter-context anchoring.
  • Director would not naturally read it.
Fit: possibly relevant for Baker engineering tracking. Not for desk-fleet operations.
Section 4 · Recommendation

The architecture, plainly stated

Three storage tiers, one messenger.

Tier 1 — Soft per-desk tasks → Obsidian vault. Tasks live as markdown files in baker-vault, organised per desk and per matter. Frontmatter carries due date, priority, owner agent, status, and links to context. Agents have full filesystem CRUD. Read at session start. Closed when the agent acts.

Tier 2 — Hard counterparty deadlines → Baker deadline system (baker_add_deadline). Externally fixed dates with counterparty consequences (Skliar 30.06.2026, Aukera covenant tests, Eastdil-owed extracts, regulatory cut-offs, residence-fee year-end). These must fire cross-session — regardless of which desk is open. Baker's existing deadline primitive handles registration, query, and nudge cadence. Surfaced by Baden-Baden Desk 12 May; folded into the architecture today.

Tier 3 — Recurring agent jobs → Baker APScheduler. Cron-like triggers ("every Monday 09:00 produce weekly status"; "monthly reconcile residence-fee ledger"). Already in production for AI Head weekly audit + Sunday digest. Fires the owning agent with the right context.

Director-facing pings come from Baker, never from the data stores. Baker reads the vault, the deadline table, and APScheduler state — decides what is worth surfacing — sends a curated Slack DM. Director does not get pinged by Obsidian. Does not get pinged by Todoist. Does not get pinged by ClickUp. Single channel: Baker → Slack DM. Cadence configurable (per-desk daily digest at 07:00; same-day push for high-priority hard deadlines; weekly Sunday roll-up).

Calendar: Google Calendar reserved for time-blocked meetings only. Agent-drafted; Director-ratified; agent creates the event.

What we drop: ClickUp as scheduling infrastructure. Existing data stays in place; new desk work stops accumulating there. Notion remains off the table until the duplicate bug is upstream-fixed.

What Director sees: a curated Slack DM from Baker at agreed cadence — the items that genuinely need his attention, with one-click links to matter context and any draft ready for ratification. Never the raw queue. Never a third-party app's native pings.

Why this is the right answer

  • It eliminates the MCP gap problem permanently. Filesystem is the API. No vendor can break delete.
  • It collocates tasks with their context. The MOVIE Desk debrief task sits one folder away from the OPERATING.md state and the talking-points pack. Context is one click away.
  • It matches the Director's stated frame. He does not attend to the data store; Baker reads the data and pings him through the one channel he already lives in (Slack). One messenger, one signal stream, zero per-app notification cruft.
  • It uses infrastructure that already exists. APScheduler is running. Vault is git-tracked. Baker → Slack DM path is in production for the Sunday digest and the AI Head weekly audit. No new system to maintain.
  • It scales across the fleet. Every desk (MOVIE, AO, Hagenauer, Baden-Baden, future) uses the same pattern. No per-desk SaaS configuration drift.

What this does not solve, and why that is fine

  • Third-party native push (Obsidian / Todoist / ClickUp). Explicitly out of scope. Baker owns the messenger role.
  • Cross-team task assignment outside the agent fleet. If Edita or Balazs ever need to be assigned a task, do it via email — not by adding them to the vault.

For AH1 — engineering eval points

Director will route this section to AH1 for engineering evaluation after Director ratification of the recommendation. The questions below are what MOVIE Desk would want answered before MOVIE Desk relies on the new architecture day-to-day.

  1. Vault folder schema. Where do tasks live? Proposed: baker-vault/_ops/agents/<desk>/tasks/YYYY-MM-DD-<slug>.md, one file per task, with frontmatter. Alternative: a single tasks.md per desk with checkbox items. Per-file is more agent-friendly (atomic writes, no concurrent-edit races). Per-file is your call.
  2. Frontmatter schema. Minimum proposed: due, priority, owner_agent, matter_slug, status, created, references. Optional: recurrence (cron string or natural-language), auto_close, blocked_by. Confirm fields + types.
  3. APScheduler integration. The vault-scanner job: how often does it run, what does it do when it finds a due task? Proposed: 06:00 UTC daily scan, posts a per-desk "due today" summary to _ops/agents/<desk>/today.md + drops Slack DM only if asked. No automatic agent dispatch on time-trigger today — desks are session-launched.
  4. Concurrent-write hygiene. Existing scar: baker-vault shared-FS race on multi-agent concurrent writes (2026-04-30 incident). Does per-file task storage solve it cleanly, or do we still need atomic single-shell commits / per-agent worktrees for the tasks directory?
  5. Task closure protocol. When an agent completes a task, what changes? Proposed: status: closed, closed_at, outcome link to produced artefact. Or move file to tasks/closed/? Affects how DataView queries are written.
  6. Obsidian Tasks + DataView plugin install. Standardising on these plugins for Director's view, or staying plugin-free and querying via grep? Plugin-free is more robust; plugins are nicer for visual scan.
  7. Recurring jobs registry. If recurring patterns live in markdown frontmatter (per-task), or in a central YAML registry, or in APScheduler code — choose one. Recommendation: YAML registry at _ops/processes/schedule-registry.yml, AH1-owned, AI Head can propose changes, Director ratifies.
  8. Migration of the one live task. Tomorrow's MOHG debrief task is currently in ClickUp. Migrate to vault now, or wait until the architecture is built? Recommendation: build the minimum vault schema today, migrate the one task, prove the pattern. ClickUp space remains as legacy until manually cleared.
  9. Baker deadline system integration. New since v1 of this doc, per Baden-Baden Desk proposal. Confirm: (a) the rule for "what counts as a hard deadline" (recommend: counterparty consequence on miss); (b) the existing nudge cadence in baker_add_deadline (lead-time before deadline, escalation pattern) and whether it needs refinement for desk-fleet use; (c) whether deadline-table state should be queryable from inside Obsidian via a daily auto-generated _ops/agents/<desk>/upcoming-deadlines.md mirror, so desks see their own deadlines at session start. AH1 to validate that the deadline channel surfacing is already routed via Slack DM (per Baker's existing primitive) — if so, no new messenger work needed.
  10. Baker Slack DM cadence + format. Director's frame: Baker is the messenger; Obsidian / Todoist / ClickUp are not. Confirm: (a) daily 07:00 per-desk "due today" digest covering both vault tasks and approaching hard deadlines, one DM per desk; (b) same-day push for high-priority items only (threshold to be defined); (c) Sunday weekly digest with overdue + upcoming items per desk. Format: bullet list with matter-slug prefix and inline Obsidian link. Existing Sunday digest infrastructure is the place to extend, not rebuild.
  11. Scope of v1. Recommendation: ship v1 with just (a) vault schema, (b) one daily scanner job, (c) one task migrated to vault, (d) one hard deadline registered in baker_add_deadline (recommend: 31.12.2026 residence fee deferral year-end as the test case for MOVIE Desk). Plugins, registry, full digest, multi-desk rollout — defer to v2 once v1 is stable.
