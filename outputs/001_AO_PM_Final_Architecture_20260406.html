The Self-Learning Loop — Every query makes the PM smarter
View Files → PM Invocation (Opus) → Q&A + Tools → Extract Insights → pm_pending_insights → Director Review → View Files
↺ the wiki compounds
Path 1: Director Debrief
Director tells Baker something new → Baker writes directly into view file → git commit.

Fast. Authoritative. No staging needed.
Path 2: Discovered Knowledge
Baker discovers insight during Q&A → extended Opus call extracts it → pm_pending_insights table → Director promotes or rejects.

Safe. No autonomous wiki corruption.
Implementation Detail — What was built
ComponentLocationWhat It Does
pm_pending_insights PostgreSQL 12-column table. pm_slug-based (generic for all PMs). Status flow: pending → approved/rejected → promoted.
_auto_update_ao_state() capability_runner.py Extended Opus call (max_tokens 700). Extracts state updates AND wiki_insights in one call. pop() before state merge.
_get_extraction_dedup_context() capability_runner.py Feeds existing pending + rejected insights (with review_note) into Opus prompt. Three-layer dedup: Opus self-dedup + rejection learning + ILIKE fallback.
_store_pending_insights() capability_runner.py Max 3 per interaction. Min 50 chars. High/medium confidence only. ILIKE dedup on 80 chars within 7 days.
_get_pending_insights_context() capability_runner.py Injects top 5 pending insights into system prompt with total count. "N more — say 'show all pending' for full list."
get_pending_insights tool agent.py Director says "show pending" → PM calls tool → returns full list from DB.
update_pending_insight tool agent.py Director says "promote #N" or "reject #N" → PM calls tool → updates status + review_note.
Briefing integration briefing_trigger.py Daily briefing shows pending count when > 0. Non-fatal if table missing.
Safety Guards
pop() before state merge
wiki_insights removed from Opus JSON before ao_project_state update. If extraction returns garbage, state update still works.
Three-layer dedup
1. Opus sees existing pending in prompt (self-dedup)
2. Opus sees rejections with reasons (learns)
3. ILIKE on first 80 chars as fallback
Quality filters
Min 50 chars (no trivial observations). High/medium confidence only. Max 3 per interaction. Low confidence dropped.
Confidence definitions
High: Director-stated or document-confirmed
Medium: Inferred from Q&A with evidence
Low: Speculative (auto-dropped)
Promote flow clarity
"Approved. Queued for Code Brisen to merge into {file}." Director knows it's not instant — view file edit is a separate deploy step.
Rejection propagation
Rejected insights + review_note fed back to Opus as "DO NOT re-extract." The PM learns from every rejection.

Four Rules
1
Wiki is gospel
If anything contradicts the view files, the view files win. Everything else is input or scratchpad.
2
Director writes directly
Debrief sessions go straight into view files. No staging needed. Baker commits immediately.
3
Baker proposes, Director approves
Discovered knowledge goes to pm_pending_insights. Baker never writes to view files autonomously.
4
The system learns from rejection
Rejected insights with reasons are fed back to the extraction prompt. The PM compounds knowledge from approvals AND rejections.
