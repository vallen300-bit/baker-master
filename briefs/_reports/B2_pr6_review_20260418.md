## 4. Nice-to-have

### N1 — `load_hot_md` empty-file behavior is undocumented

Test `test_load_hot_md_empty_file_returns_empty_string` confirms: empty file → `""` (not `None`). But the docstring only says:

> *"Returns: File contents as str, or None if the file does not exist (valid zero-Gold state per CHANDA Inv 1)."*

Callers see two zero-Gold signals: `None` (missing) and `""` (empty). The Step 1 builder must use `if not content:` to capture both — `if content is None:` would miss the empty case.

**Fix.** One line in the docstring: *"Returns None for missing file or '' for empty file — both represent zero-Gold per Inv 1; callers should treat both as 'no hot-list available'."* Cosmetic. Prevents a future Step 1 author from writing `if content is None:` and silently dropping empty hot.md cases.

### N2 — `render_ledger` doesn't truncate long detail lines

A `director_note` or payload-summary can be arbitrarily long. `_collapse` handles whitespace but not length. A 1000-char payload renders as a 1000-char line.

For Phase 1 with Director-curated notes (typically <200 chars) this is fine. If we ever surface auto-generated payloads in the ledger (e.g., Step 6 cross-link writes recording the entire wiki frontmatter), lines could bloat.

**Fix.** Either (a) defer until observed (current state — reasonable), or (b) preemptively cap detail at e.g. 200 chars with `…` ellipsis. Phase 1 doesn't need this.

### N3 — `_LEDGER_COLUMNS` is an implicit schema contract with the migration

```python
_LEDGER_COLUMNS = (
    "id", "created_at", "action_type", "target_matter",
    "target_path", "signal_id", "payload", "director_note",
)
```

This tuple shape is locked to `migrations/20260418_loop_infrastructure.sql`. If a future migration adds a column (e.g., `ledger_source` to distinguish UI vs API writes), `load_recent_feedback` silently misses it, AND the `dict(zip(...))` produces `{key: value}` pairs that mismatch when `cursor.fetchall()` returns more values than expected (extra row values get dropped).

**Fix.** Three options:
- **(a)** Add a comment: `# MUST stay in sync with migrations/20260418_loop_infrastructure.sql feedback_ledger schema`. Cheap.
- **(b)** Use `cur.description` to read column names dynamically. Robust to schema changes but loses static contract.
- **(c)** Define a `LedgerRow` Pydantic / dataclass model and validate post-fetch. Heavier; probably overkill for Phase 1.

I lean (a). The coupling is real but small surface; a sync comment + post-merge update of `_LEDGER_COLUMNS` when schema changes is cheap.

### N4 — `load_recent_feedback` rollback failure is silently swallowed

```python
try:
    conn.rollback()
except Exception:
    pass
```

The test `test_load_recent_feedback_db_error_tolerates_rollback_failure` confirms the original error still surfaces, but the rollback exception is gone. If the rollback failure is ever interesting (e.g., signals a deeper connection problem), it's invisible.

**Fix.** Optional: wrap in `try / except Exception as inner: logger.warning("rollback failed: %s", inner)`. Adds a logger import and a one-liner. Phase 1: defer — current behavior is correct (don't shadow original error), just opaque on the secondary failure. Worth a NICE flag for KBL-C observability if `LoopReadError`s start appearing without explanation.

### N5 — No live-PG round-trip test for `load_recent_feedback`

All ledger tests are mock-based (psycopg2 `MagicMock`). Mocks catch logic bugs but not actual SQL syntax errors against PG (e.g., a future column rename in the SQL statement would pass mocks but fail in production).

**Fix.** Mirror the `@requires_db` pattern from `test_migrations.py`: one round-trip test that creates a feedback_ledger row + reads it via `load_recent_feedback`. Skipped without `TEST_DATABASE_URL`. ~30 lines. Not a v1 blocker — mock coverage is solid — but worth adding when PG-test infrastructure is wired more broadly.

## 8. Pattern observations

### Difference from `kbl/slug_registry.py` and `kbl/layer0_rules.py`

These prior modules are **loaders** with caching (lock + module-level cache + reload). `kbl/loop.py` is a **stateless helper module** — no cache, no lock, no `reload()`. Correct departure: per CHANDA Leg 3, hot.md and feedback_ledger are read **fresh per Step 1 invocation** (the loop's whole point is that Director changes propagate immediately). Caching would defeat the architecture.

### Connection ownership

`load_recent_feedback(conn, ...)` takes the connection as an argument; caller owns lifecycle. Clean separation — no global pool dependency, no implicit transaction scoping. Step 1 caller decides whether to share a connection across helpers or open per-call.

### Error class consistency

`LoopReadError(RuntimeError)` matches the established pattern (`SlugRegistryError(RuntimeError)`, `Layer0RulesError(RuntimeError)`). Three loaders, three exception classes, same superclass — predictable for upstream `try / except RuntimeError` catches.
