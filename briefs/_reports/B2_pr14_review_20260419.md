## Nice-to-have (N, deferrable)

### N1 — Daily-cap query LHS is session-TZ-dependent

**File:** [kbl/cost_gate.py:143-147](kbl/cost_gate.py:143)

```sql
WHERE ts::date = (NOW() AT TIME ZONE 'UTC')::date
```

`ts` is `TIMESTAMPTZ` ([memory/store_back.py:6516](memory/store_back.py:6516)). `ts::date` on a TIMESTAMPTZ uses the **session timezone** to derive the date; the RHS explicitly converts to UTC first. In Render/Neon's default UTC session, both sides agree — in production this works and the index `idx_cost_ledger_day ((ts::date))` matches. But under a non-UTC session (local dev, future migration, a `SET timezone = ...` anywhere upstream), rows around the UTC-midnight boundary are miscounted, silently under-charging the cap.

Brief §6 (point 6): "UTC boundary is critical (Director ratified UTC day). Check the timestamp arithmetic — `datetime.now(UTC).date()` or equivalent, not local-time." The RHS satisfies this; the LHS inherits session-TZ fragility.

**Two-option fix:**
- (a) `WHERE (ts AT TIME ZONE 'UTC')::date = (NOW() AT TIME ZONE 'UTC')::date` — symmetric UTC cast. One SQL line.
- (b) Brief's original form: `WHERE ts >= date_trunc('day', NOW() AT TIME ZONE 'UTC') AT TIME ZONE 'UTC'` — no ::date cast, uses the index via range scan.

Both zero-risk. Not blocking because Render session is UTC.

### N2 — Circuit-breaker probe machinery is dead

**File:** [kbl/cost_gate.py:209-243](kbl/cost_gate.py:209) `_is_circuit_open`.

Schema has `last_probe_at` ([migrations/20260419_step5_signal_queue_opus_draft.sql:42](migrations/20260419_step5_signal_queue_opus_draft.sql:42)), and `_is_circuit_open` has cooldown-aware branching that returns False once `now >= last_probe_at + probe_interval_sec`. But no code path bumps `last_probe_at` pre-call — only `record_opus_success` (requires a successful fire, which the open circuit blocks) and `reset_opus_circuit` (operator). Net effect: **once tripped, the circuit never self-heals**; the cooldown logic is unreachable.

Module docstring claims "inline probe during `can_fire_step5()` is acceptable — we do not schedule a separate cron here." The inline probe is not implemented.

**Two-option fix:**
- (a) Implement the probe: on a call that meets `opened_at + probe_interval`, bump `last_probe_at = NOW()`, allow one call through, let `record_opus_success / _failure` resolve the state. One ~15-line branch.
- (b) Delete the dead cooldown logic + `last_probe_at` probe semantics; document "operator reset only" as the recovery path. Simpler, but blocks auto-healing on transient Anthropic outages.

Brief §3 said "minimal inline version acceptable" — both options fit. Not blocking because manual `reset_opus_circuit()` exists.

### N4 — Commit-before-raise silent error swallow

**File:** [kbl/steps/step5_opus.py:687-689, 716-719, 745-748](kbl/steps/step5_opus.py:687)

```python
try:
    conn.commit()
except Exception:
    pass
```

Three occurrences: unknown-decision path, cost-gate-denial path, R3-exhaust path. The bare `except Exception: pass` hides a commit failure. The point of commit-before-raise is terminal-state durability — if the commit *itself* fails, the terminal-state flip is silently lost AND the operator gets no telemetry signal.

**Fix:** log the exception (`logger.warning("commit-before-raise failed: %s", e)`). Zero behavioral change, but now an operator sees "we intended to flip to opus_failed but the commit also failed — the caller's rollback may drop it."

### N5 — Cost-ledger row count per call deviates from brief

Brief point 11: *"Verify cost ledger row count: 0 on stub paths, 1 on full synthesis (success or final R3 fail)."*

Impl writes **one row per R3 attempt** ([kbl/steps/step5_opus.py:590, 603, 615](kbl/steps/step5_opus.py:590)) — including failed attempts (cost_usd=0, success=False). On R3 exhaust: 3 rows. Test `test_r3_retry_ladder_exhausted_raises_and_marks_failed` asserts `len(cost_rows) == 3` — this is a deliberate design choice, documented in `_fire_opus_with_r3` docstring: "Every attempt — successful or failed — writes a kbl_cost_ledger row so the Director can audit retry cost."

Cost-cap math is preserved (failed rows contribute cost_usd=0 to `SUM(cost_usd)`). Audit story is richer (operator can see retry latency + token counts per attempt). I'd ratify the deviation, but flag to AI Head for an explicit "accept design choice" or "roll to 1-row aggregate".
