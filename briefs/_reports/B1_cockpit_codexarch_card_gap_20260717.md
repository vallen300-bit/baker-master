## Root cause

`generate_cockpit_layout.py::build()` derived the card set as:

```python
driveable = set(ports)                                   # tmux launch manifest (terminal-* only)
app_claude = {s for s,a in registry.items() if runtime == "app-claude"}
card_slugs = driveable | app_claude
```

codex-arch has `runtime = app-codex` (registry line ~155, active + bus_enabled, shared-specialist). It is:
- **not** in `driveable` — the launch manifest's eligibility filter is `status active AND runtime startswith "terminal-"`, so an `app-codex` seat is correctly excluded (it has no tmux/ttyd terminal to drive);
- **not** in `app_claude` — that set matched `== "app-claude"` exactly, so `app-codex` missed it.

→ codex-arch fell into neither set → `generate_cockpit_layout.py` never emitted a card. The manifest omission the dispatch pointed at is *correct* (it is not a terminal seat); the real defect is the layout generator hard-coding the status-only app-seat category to `app-claude` rather than any desktop-app runtime.

## Fix (at source, not hand-edited manifest)

- Broadened app-seat detection: `runtime.startswith("app-")` (covers `app-claude`, `app-codex`, and any future `app-*`).
- Renamed the misleading `app_claude` card field + count → `app_seat` across the generator, the page JS (`cockpit.js`: 4 refs), and the layout test — so a future `app-*` runtime is not mistaken for a terminal seat.
- Files: `scripts/generate_cockpit_layout.py`, `scripts/cockpit_static/cockpit.js`, `scripts/cockpit_static/cockpit_layout.json` (regenerated), `tests/test_cockpit_layout.py`.
