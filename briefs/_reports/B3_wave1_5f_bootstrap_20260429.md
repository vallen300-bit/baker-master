### Unexpected slug mutations at step 3 — none

### Track 5c converter slug-field quirks visible in the live output (NOT 5f scope)

These reflect parser limitations in `triaga_to_priorities.py`. The `_priorities.yml` shape is schema-valid and the regen accepts it, but a few `slug:` values landed mis-parsed:

1. **Em-dash split bug — Q17, Q18, Q37.** Header lines like `**Q17 — [private-assets — slug TBD] — Barclays UK mortgage renewal form issued**` get split at the *first* ` — ` by `_HEADER_LINE_RE` (non-greedy). Result: `slug: '[private-assets'` (with leading bracket) + `description: 'slug TBD] — Barclays UK …'`. Visible in hot.md as `**[private-assets**: slug TBD] — …`. Affects Q17, Q18, Q37.
2. **`/` always-split — Q30, Q31.** `Q30 — tax / lana — …` → `slugs: [tax, lana]` (the `tax /` was a category-prefix in the export, not a real slug pair). `Q31 — tax / cbp — …` → `slugs: [tax, cbp]` (canonical was `us-cbp`). Q31 is dismissed-only, so the slugs land in the `dismissed:` block.
3. **Non-canonical slugs emitted but absent from `slugs.yml`** — `orbit`, `amir` (Q41); `tax`, `lana` (Q30); `cbp` (Q31); `[private-assets`, `[philippe-soulier` (header-split artefacts). The regen renders these in hot.md verbatim — no validation against the registry happens at the converter or regen layers.

These do **not** block this PR's purpose (proving the pipeline runs end-to-end). They surface real Track 5c follow-ups before live cycles route signals on these matters.

**Suggested 5c follow-up brief:**
- Harden `_HEADER_LINE_RE` to handle em-dash-bearing slug fields (e.g. greedy-then-anchored on `**` or detect a ` — slug TBD]` sentinel before the header split).
- Add an opt-in registry-allowlist filter (`--registry slugs.yml`) so unknown slugs raise a converter warning instead of writing them silently.
- Stop splitting on `/` when the left token equals a known category name (`tax`, `legal-risk`, etc.) — treat as prefix, not slug.
