## 3. Should-fix

### S1 — `version` type mismatch with slug_registry + B3's Step 0 draft

**Location:** `kbl/layer0_rules.py:121-124`.

```python
version = data["version"]
if not isinstance(version, str) or not version.strip():
    raise Layer0RulesError(
        f"{path}: `version` must be a non-empty string (got {version!r})"
    )
```

**Issue.** Three different `version` conventions are now in play across the project:

| Source | `version` type | Example |
|---|---|---|
| `kbl/slug_registry.py` | `int` | `version: 1` |
| `baker-vault/slugs.yml` (live) | int | `version: 1` |
| B3's `KBL_B_STEP0_LAYER0_RULES.md` draft (commit `6341b94`) | int | `version: 1` |
| **PR #4 fixture + loader** | **string** | `version: "1.0.0"` |

If B3 commits `baker-vault/layer0_rules.yml` with `version: 1` per their draft, B1's loader **rejects with `Layer0RulesError`** — the production vault loader can't load the production rules file.

**Fix options:**
- **(a)** Loader accepts both: `if not isinstance(version, (str, int))`. Cheap. Consumers get whatever YAML provided.
- **(b)** Pick string-semver as canonical for Layer 0 (more flexible: `"1.0.0"` → `"1.1.0"` for additive rule changes vs `"2.0.0"` for breaking schema changes), document the divergence from slug_registry's int convention.
- **(c)** Pick int to match slug_registry; loader changed to `isinstance(version, int)`; fixture updated to `version: 1`.

I lean **(c)** — vault-level convention consistency. Two YAMLs in the same vault using different version-format conventions is operator-confusing. Semver-string flexibility isn't load-bearing for a rules file (we already version control via git history).

Easy to apply inline: 3-line patch (loader assertion + fixture YAML + test assertion).

### N2 — Two env-var conventions for two loaders in the same vault

`slug_registry` uses only `BAKER_VAULT_PATH` (test isolation via fixture-directory `monkeypatch`). `layer0_rules` adds `KBL_LAYER0_RULES_PATH` for explicit-file override (test isolation via direct file pointer).

Both work; both are validated by tests. But:
- Two patterns for "test override" across two loaders is mildly inconsistent
- Operator question: "if I want to point Mac Mini at a custom layer0_rules file, which env var?" — answer is `KBL_LAYER0_RULES_PATH`, but slug_registry has no equivalent

**Fix.** Either add `KBL_SLUGS_PATH` to `slug_registry` (symmetric), or document the divergence in `kbl/__init__.py` so the convention is discoverable. Lean: symmetric — add `KBL_SLUGS_PATH` to slug_registry in a follow-up PR.

### N3 — PR #4 ships loader only; production rules file is a separate prereq

The loader is production-ready, but `baker-vault/layer0_rules.yml` doesn't exist yet. Step 0 evaluation can't be wired until that file is committed to baker-vault.

This is **not a PR #4 issue** — the loader's job is to load whatever YAML is at the path. Just flagging that "Layer 0 functional" is gated on B3's vault commit, not on PR #4 merge.

### N4 — `Layer0Rule.match` is mutable inside a frozen dataclass

```python
@dataclass(frozen=True)
class Layer0Rule:
    name: str
    source: str
    match: dict[str, Any]   # ← reference is frozen, dict contents are not
    detail: str
```

`r.match["new_key"] = "value"` is allowed even though the dataclass is `frozen=True`. Pure paranoia — no consumer is supposed to mutate `match` post-load — but a defensive `types.MappingProxyType(match)` cast in `_parse_yaml` would make it actually immutable. Cost: 1 line, marginal benefit. Defer unless a future bug surfaces.

### N5 — `_cache_source` global declaration is broader than needed

```python
def load_layer0_rules(path: ...) -> Layer0Rules:
    global _cache, _cache_source
    ...
```

The explicit-path branch doesn't write `_cache_source` (correctly — explicit paths bypass cache). The `global` declaration could be scoped only to the default-path branch. Cosmetic; current shape works.
