## BLOCKERS (must fix)

### B1 — LaunchAgent environment missing secrets (`DATABASE_URL`, `ANTHROPIC_API_KEY`, `QDRANT_*`, `VOYAGE_API_KEY`)

**Severity:** High — the pipeline will import-fail on the first real tick after go-live.

**Where:** [`launchd/com.brisen.kbl.pipeline.plist:18-22`](../../launchd/com.brisen.kbl.pipeline.plist) and the sibling plists + [`scripts/kbl-pipeline-tick.sh`](../../scripts/kbl-pipeline-tick.sh).

**What's wrong:**
- `kbl/db.py:27` reads `os.environ["DATABASE_URL"]` (bare subscript, not `.get()`).
- The LaunchAgent plist sets only `PATH` under `EnvironmentVariables`. No `DATABASE_URL`, no `ANTHROPIC_API_KEY`.
- The wrapper script does NOT source `~/.zshrc` (or any other env file).
- macOS `launchd` does **not** source login shells (`.zshrc`) when starting a LaunchAgent. The plist comment ([§8 of the brief, line 658-659](../KBL-A_INFRASTRUCTURE_CODE_BRIEF.md)) says *"Secrets loaded from `~/.zshrc` via login shell"* — that assumption is incorrect for LaunchAgent execution.

**Effect:**
1. `KBL_FLAGS_PIPELINE_ENABLED=false` path still works — the shell-level flag check in `kbl-pipeline-tick.sh:58-61` fires before any Python is invoked. ✓
2. Once Director flips the flag true: `python3 -m kbl.gold_drain` (Step 4) runs `import kbl.db` → `KeyError: 'DATABASE_URL'` → exit 1. Wrapper catches nonzero exit and logs `WARN: gold_drain exited nonzero`. Step 5 (`pipeline_tick`) fails the same way. Every tick logs two WARNs and accomplishes nothing.
3. `emit_critical` from the `git pull` conflict path ([line 38](../../scripts/kbl-pipeline-tick.sh)) also fails the same way — we lose the escalation channel silently.

**Fix options (in descending order of my preference):**

1. **Dedicated env file sourced by the wrapper.**
   ```bash
   # kbl-pipeline-tick.sh top
   [ -f "${HOME}/.kbl.env" ] && . "${HOME}/.kbl.env"
   ```
   Install script creates `~/.kbl.env` with `chmod 600`, Director populates on first deploy. Clean, plist-free, auditable, no shell-startup side effects.

2. **`launchctl setenv` at install time.** Install script prompts Director for each secret, runs `launchctl setenv DATABASE_URL "$VALUE"` etc. Values persist only until logout — a reboot restores them only if the install step re-runs. Fragile.

3. **Plist `EnvironmentVariables` injection at install.** Install script reads `~/.kbl.env`, patches all 4 plists with the values, then `launchctl load`. Works but secrets end up in `~/Library/LaunchAgents/*.plist` which is user-readable — probably unacceptable for `ANTHROPIC_API_KEY`.

4. **Source `~/.zshrc`.** Simplest but `.zshrc` is a zsh config file, not a pure env file — `source ~/.zshrc` in a bash wrapper fires interactive hooks, aliases, etc. Fragile.

Recommended: option 1 + brief §6 update so future onboarding doesn't re-hit this.

**Repro prediction:** on a fresh macmini install with only `PATH` in the plist, the very first `launchctl start com.brisen.kbl.pipeline` (or the first `StartInterval` tick after `launchctl load`) will write `KeyError: 'DATABASE_URL'` to `/var/log/kbl/pipeline.stderr`.

## SHOULD FIX (strongly recommended)

### S1 — Gold drain: lock release + concurrent-drain race

**Where:** [`kbl/gold_drain.py:54-63`](../../kbl/gold_drain.py).

**What:** `drain_queue()` opens a transaction, does `SELECT ... FOR UPDATE SKIP LOCKED`, then calls `conn.commit()` immediately to release the row locks. The comment says "Mac Mini is sole consumer" — but [`scripts/kbl-gold-drain.sh`](../../scripts/kbl-gold-drain.sh) is also installed as a standalone command and the brief describes it as "useful for ad-hoc flush". Two drainers can therefore race:

- Drain A claims rows {1,2,3}, commits (releases locks), applies filesystem writes, commits + pushes.
- Drain B (ad-hoc, concurrent) claims rows {1,2,3} too, sees `author: director` already set by A's filesystem write, returns `noop` for each. promoted_paths is empty → skips commit+push. Marks PG rows {1,2,3} as done.
- Drain A's push then FAILS (network blip) → rolls back filesystem via `checkout HEAD -- paths`. PG rows are now `processed_at=NOW()` but filesystem has been reverted. State drift.

**Mitigations ordered by cost/benefit:**
1. Keep the claim transaction open until after push succeeds. Commit only once (the single UPDATE to `processed_at`). This keeps SKIP LOCKED's guarantee through the whole critical section. Minor refactor.
2. Add a drain-level advisory lock (`pg_try_advisory_lock(<const>)`) at the top of `drain_queue()`. Second drain caller exits silently. Matches the shell-level flock pattern.
3. Leave as-is and document that ad-hoc drain must never be run while the LaunchAgent is enabled. Riskiest.

### S2 — Cost-alert date boundary mismatch (Europe/Vienna vs UTC)

**Where:** [`kbl/cost.py:197`](../../kbl/cost.py) uses Python `date.today().isoformat()` for the dedupe key, but [`kbl/cost.py:119-120`](../../kbl/cost.py) uses DB `NOW()::date` for the spend sum. The brief's D11 (§16 env var table) specifies system TZ `Europe/Vienna` while DB is UTC.

**Effect:** During the 1-2 hour window after UTC midnight but before Europe/Vienna midnight (01:00-02:00 local, depending on DST):
- Python `date.today()` still returns the prior local date.
- DB `NOW()::date` has already rolled to the new UTC date.
- The spend sum is the NEW day's partial spend (small). Python's dedupe key is the OLD day. If the 80% alert fired at 23:00 local = 22:00 UTC (still old UTC day, old dedupe key), and at 00:30 UTC the cap is hit again at 101% of the cap from the remaining prior-day alerts... the 100% threshold check (`today = 0.02, cap = 15, today_pct = 0.1%`) is fine — the scenario doesn't actually fire. So the real risk is an ALERT MISS in the inverse direction: if cap was hit at 22:00 UTC (100% fired under OLD UTC date), `daily_cost_circuit_clear` at 03:15 Europe/Vienna (02:15 UTC) cleared circuit prematurely... actually this is also OK because the purge runs at 03:15 local Europe/Vienna regardless of UTC date.

Re-reading: the hazard is more subtle. `_try_dedupe(alert_key)` uses `cost_100pct_<python_date>`. If Python and DB disagree on "today" across midnight, the same CALENDAR day could fire two `100pct` alerts with two different keys. Low probability (requires cap-exceeded to happen exactly once per local day AND that moment straddling UTC midnight) but real.

**Fix:** unify on one clock. Simplest — use `datetime.now(timezone.utc).date().isoformat()` for the dedupe key so it aligns with `NOW()::date`. One line.

### S3 — `call_gemma_with_retry` doesn't short-circuit when Qwen is already active

**Where:** [`kbl/retry.py:118-152`](../../kbl/retry.py).

**What:** During a Qwen-active period (post-swap, before recovery), every new call starts at Gemma attempt 1. If Gemma is genuinely down, each signal generates **3 failing Ollama calls + 3 WARN log rows** before reaching the Qwen path. At 1 signal/minute that's 4,320 wasted calls/day + 4,320 `retry_gemma` WARN rows in `kbl_log`.

**Also:** the opposite direction — if Gemma RECOVERS during a Qwen-active period and attempt 1 succeeds, the code returns the result but leaves `qwen_active=true` unchanged. `maybe_recover_gemma()` only fires on count/hours triggers; a natural recovery signal is silently dropped.

**Fix (both issues, ~8 lines):**
```python
def call_gemma_with_retry(signal, prompt_template):
    qwen_already_active = get_state("qwen_active") == "true"
    if qwen_already_active:
        # Skip straight to attempt 4; opportunistically try Gemma
        # once every N ticks to detect recovery (or rely on maybe_recover_gemma).
        return _qwen_attempt(signal, prompt_template, fallback)
    # else run attempts 1-3 as today, then swap on exhaustion
    # On ANY successful attempt 1-3, if qwen_active was ever set,
    # trigger the recovery path in-line (not just count/hours).
```

### S4 — Gold drain: `git add` / `git commit` failure not caught

**Where:** [`kbl/gold_drain.py:175-198`](../../kbl/gold_drain.py).

**What:** `_commit_and_push` uses `check=True` on `git add` (line 176) and `git commit` (line 184). If either raises `subprocess.CalledProcessError`, the `try/except GitPushFailed` in `drain_queue` (line 83-95) does NOT catch it — the exception bubbles up, leaving:
- Filesystem modified (promote_one already wrote `author: director`).
- No commit created.
- PG rows still pending.

Next drain run sees `author: director`, returns `noop` → `promoted_paths` empty → skips commit+push → marks PG rows done. Filesystem is in the expected state (since promote_one had already written it), so in THIS case we self-heal. But if a future refactor ever splits the filesystem-write and the commit by more than one drain cycle, we'd have silent drift.

**Fix:** broaden the try/except in `drain_queue`:
```python
except (GitPushFailed, subprocess.CalledProcessError) as e:
    ...
```
Or catch `CalledProcessError` inside `_commit_and_push` and convert to `GitPushFailed` with context. One-line tweak.

### S5 — `check_and_clear_anthropic_circuit` model string is bare alias

**Where:** [`kbl/retry.py:105`](../../kbl/retry.py) — `model="claude-haiku-4"`.

**What:** Anthropic's API accepts the short alias and resolves to the latest matching version, but this behavior has historically been inconsistent across SDK versions. If Anthropic ever deprecates bare `claude-haiku-4` (as they did with pre-4.x aliases), the health check silently stops working and the circuit never auto-clears.

**Fix:** add `KBL_CIRCUIT_HEALTH_MODEL` env var with default `"claude-haiku-4-5"` (current GA version per CLAUDE.md), read via `cfg("circuit_health_model", "claude-haiku-4-5")`.

## NICE TO HAVE (optional)

### N1 — Anthropic-circuit-open WARN log has no dedupe

[`kbl/pipeline_tick.py:53-58`](../../kbl/pipeline_tick.py) writes a WARN row to `kbl_log` every tick (120s) while circuit is open. A 1-hour outage = 30 rows. Consider either local-logger-only for this specific message, or route through `kbl_alert_dedupe` with a 15-min bucket.

### N2 — `_parse_frontmatter` silently fabricates frontmatter on headerless files

[`kbl/gold_drain.py:156-163`](../../kbl/gold_drain.py): a wiki file without a `---\n...\n---\n` header gets an empty dict, then `promote_one` injects `{"author": "director", "author_verified_at": ...}` and prepends it. The intent of Gold promotion is almost certainly "mark an existing Director-reviewed wiki page", not "create frontmatter on pages that never had any". Consider returning `"error:no_frontmatter"` instead, mirroring the `error:file_not_found` pattern.

### N3 — `_commit_and_push` hardcodes branch `main`

[`kbl/gold_drain.py:206`](../../kbl/gold_drain.py) — `git push origin main`. If baker-vault ever changes default branch (unlikely but possible) this breaks silently. Read `cfg("vault_branch", "main")` instead.

### N4 — `was_inserted` return-type assumption

[`kbl/logging.py:141-145`](../../kbl/logging.py) — `bool(cur.fetchone()[0])`. psycopg2's default boolean adapter returns Python `True`/`False`, but B1 correctly flagged this for live-DB verification. Worth a first-deploy check: psql `SELECT (xmax = 0) FROM kbl_alert_dedupe LIMIT 1` to confirm `True`/`False` not string `'t'`/`'f'`. If it came back as `'f'`, `bool('f')` is `True` (non-empty string) and every CRITICAL fires. Defensive fix: `cur.fetchone()[0] is True`.

### N5 — Brief §14 acceptance line 1610 says "3 new columns" but §5 + implementation add 4 (incl. `started_at`)

Doc drift inside the brief. Not a B1 problem; flag for AI Head's own hygiene pass.

### N6 — No venv / pip-install step in the install script

The install script checks for `yq`, `flock`, `ollama` but never verifies `psycopg2-binary`, `anthropic`, `yaml`, `requests` are importable. If a fresh Mac Mini skips `pip install -r requirements.txt`, first tick gets `ModuleNotFoundError`. Either add a `python3 -c "import psycopg2, anthropic, yaml, requests"` sanity check to the install script, or document the venv-activation prereq in the install script's preamble.
