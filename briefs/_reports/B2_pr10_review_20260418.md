## 3. Should-fix

### S1 — `status` column writes violate the live KBL-A CHECK constraint

**Scope:** **pipeline-wide** (PR #7 / #8 / #10 all share this pattern). Not a PR #10 design bug; inherited from the §3.2 status-values-vs-§5.3-mirror-values coordination gap.

**Location:** [`kbl/steps/step2_resolve.py`](../../kbl/steps/step2_resolve.py) lines 34-36, 92-97, 108-115:

```python
_STATE_RUNNING = "resolve_running"
_STATE_NEXT = "awaiting_extract"
_STATE_FAILED = "resolve_failed"
# ...
cur.execute("UPDATE signal_queue SET status = %s WHERE id = %s", (_STATE_RUNNING, signal_id))
```

**The issue.** The live `signal_queue.status` CHECK constraint (from KBL-A at [`memory/store_back.py:6408-6414`](../../memory/store_back.py)) is:

```sql
CHECK (status IN (
    'pending','processing','done','failed','expired',
    'classified-deferred','failed-reviewed','cost-deferred'
))
```

PR #10's writes of `'resolve_running'`, `'resolve_failed'`, `'awaiting_extract'` are **all** outside this set. On any real PG signal, the UPDATE will raise `CheckViolation` and the dispatcher crashes.

**Why tests don't catch it.** All 41 new tests mock `psycopg2` via `MagicMock` (see `tests/test_step2_resolve.py:MockCursor`). The CHECK constraint is PG-side; no live-PG round-trip exercises `_mark_running` / `_write_result`, so the constraint never fires during CI.

**This is pipeline-wide.** PR #8 (STEP1-TRIAGE-IMPL @ `pr8-verify`) writes `status='triage_running'` (line 381 of `kbl/steps/step1_triage.py`) — same CHECK violation. PR #7 (LAYER0-IMPL) writes `status='dropped_layer0'` via `_process_layer0` — also not in the CHECK set. Three step impls inherit the same gap.

**Brief design intent.** KBL-B §5.1 ratified a **two-track migration**:
- Legacy `status` CHECK stays at the 8 KBL-A values
- New `stage` + `state` columns added (§5.2 schema)
- KBL-B writes BOTH, with `status` mapped through §5.3 compatibility mirror
- After Phase 2 burn-in, `status` dropped

None of this migration has shipped. No `stage` / `state` columns in any migration file. No §5.3 mirror writes in step impls.

**Three remediation paths** (AI Head call):

- **(a) Ship the §5.1 two-track migration now** — add `stage` + `state` columns per §5.2; rework PR #7 / #8 / #10 to write both columns with `status` mirrored per §5.3. Faithful to brief; highest coordination cost. Estimated: 1 migration PR + 3 impl touches.

- **(b) Ship a `status` CHECK expansion migration** — add all 24 KBL-B status values (§3.2 list) to the CHECK set. PR #7 / #8 / #10 ship as-is. Deferrable: two-track migration becomes Phase-2-burn-in cleanup per §5.7. Cheapest; deviates from §5.1.

- **(c) Rework PR #10 (+ #7 + #8) to write mirror values** — use `'pending'` / `'processing'` / `'done'` / `'failed'` per §5.3 table. Loses per-step observability granularity (every step looks like `processing`). No migration needed. Not recommended — defeats the point of per-step states.

Lean **(b)** for Phase 1 speed + (a) for Phase 2 cleanup. (b) gives you one migration PR that unblocks all three step impls; (a) costs the coordination but is what the brief actually ratified.

**Critical:** PR #10 is **not independently mergeable** to main until one of (a)/(b) lands. Same applies to PR #7 and PR #8.

**This is a deploy-sequence gap, not a code-design bug.** PR #10's code correctly follows brief §3.2 values; the CHECK-expansion to match didn't ship alongside. Flagging as should-fix because the remediation isn't necessarily in PR #10 itself — AI Head decides the pipeline-wide path.

**Consistency note.** While the brief's §3.2 list uses mixed naming (`triaging`, `resolving`, `extracting`, `opus_running`, `finalize_running`, `committing`), PR #10 uses `resolve_running` (the `<step>_running` shape matching `opus_running` + `finalize_running`). ## 4. Nice-to-have

### N1 — Missing `VOYAGE_API_KEY` is not degraded-mode

**Location:** [`kbl/voyage_client.py:67-71`](../../kbl/voyage_client.py) + [`kbl/resolvers/_embedding.py:184-197`](../../kbl/resolvers/_embedding.py).

`voyage_client.embed()` raises `ValueError` when `VOYAGE_API_KEY` is unset:
```python
if not key:
    raise ValueError(f"{_API_KEY_ENV} env var not set and no api_key passed")
```

`_embedding.py` only catches `VoyageUnavailableError` in its degraded-mode handler:
```python
try:
    query_vec = embed_fn(text)
except VoyageUnavailableError:
    # degraded-mode: empty paths + success=False cost row
```

`ValueError` propagates up → dispatcher's `except Exception` catches → marks signal `resolve_failed` + raises `ResolverError`.

**Problem.** Missing API key is a CONFIG issue (same class as a 401 response, which IS caught as `VoyageUnavailableError`). Treating it differently creates asymmetric behavior:
- 401 from Voyage → degraded-mode (empty paths + cost row with success=False) → signal continues
- Missing key → `resolve_failed` → signal terminal

A misconfigured environment (forgotten API key rotation, accidentally cleared env) would halt all transcript/scan signals with `resolve_failed` rather than degrade-and-continue. Email/WA would keep working, creating a subtle partial-pipeline-outage mode.

**Fix.** In `_embedding.py`, catch `ValueError` alongside `VoyageUnavailableError` in the embed call, OR wrap the embed call so missing-key surfaces as `VoyageUnavailableError`:
```python
try:
    query_vec = embed_fn(text)
except (VoyageUnavailableError, ValueError):
    # degraded-mode
```

Trivial. Makes config failures observable via the same cost-ledger `success=False` pattern as transient failures. Operator sees consistent audit trail regardless of which class of Voyage failure fires.

### N2 — WhatsApp sliding window docstring doesn't specify which timestamp

**Location:** [`kbl/resolvers/whatsapp.py:1-11`](../../kbl/resolvers/whatsapp.py).

Docstring: *"Matches signals in the same `chat_id` within a 90-day sliding window."*

Implementation uses `sent_at` (if present) OR `now()` as the **anchor**, and `created_at` (ingestion time) as the **filter column**. For live WA signals (ingested near-real-time), `sent_at ≈ created_at` and the distinction is immaterial. For back-filled / historical signals (ingested long after they were sent), the two diverge.

**Example.** A WhatsApp signal with `sent_at=2026-01-01` ingested on `created_at=2026-04-15`:
- Anchor = sent_at = 2026-01-01
- Window = 2025-10-03 to 2026-01-01 on `created_at`
- But `created_at=2026-04-15` is NOT in that window → this signal itself wouldn't resolve to a prior thread in its own chat_id

The filter is on PRIOR signals' `created_at`, so the self-exclusion isn't the real concern — but the window semantic is "signals ingested within 90d of this signal's sent_at" rather than "signals that were sent within 90d of this signal's sent_at."

**Fix.** Either:
- **(a)** Document the ingestion-vs-sent-time distinction in the resolver docstring: "Window anchored on `sent_at` (or `now()` fallback), filtered on PRIOR signals' `created_at`. For conversation-time proximity use `payload->>'sent_at'` filtering — tradeoff: requires index."
- **(b)** Filter on `payload->>'sent_at'` for true conversation-time proximity. Requires index on `((payload->>'sent_at')::timestamptz)` for performance.

Lean (a) for now — live ingestion makes the divergence negligible. Revisit if Phase 2 surfaces back-fill use cases.

### N3 — `KBL_STEP2_RESOLVE_THRESHOLD` accepts out-of-range values silently

**Location:** [`kbl/resolvers/_embedding.py:33-41`](../../kbl/resolvers/_embedding.py).

```python
def _resolve_threshold() -> float:
    raw = os.environ.get(_THRESHOLD_ENV)
    if not raw:
        return _DEFAULT_THRESHOLD
    try:
        parsed = float(raw)
    except ValueError:
        return _DEFAULT_THRESHOLD
    return parsed
```

A misconfigured `KBL_STEP2_RESOLVE_THRESHOLD=2.0` would silently prevent any match (cosine similarity ∈ [-1, 1]; threshold 2.0 unreachable). `KBL_STEP2_RESOLVE_THRESHOLD=-1.0` would match everything including orthogonal vectors.

**Fix.** Clamp to [0, 1] with WARN + fallback to default:
```python
if not 0.0 <= parsed <= 1.0:
    logger.warning("KBL_STEP2_RESOLVE_THRESHOLD=%s out of [0,1]; using default %s", raw, _DEFAULT_THRESHOLD)
    return _DEFAULT_THRESHOLD
```

3 lines + 1 test. Closes a subtle misconfiguration footgun.

### N4 — `_validate_paths` accepts `wiki/../etc/passwd`

**Location:** [`kbl/steps/step2_resolve.py:147-161`](../../kbl/steps/step2_resolve.py).

```python
if not isinstance(p, str) or not p.startswith(_WIKI_PREFIX):
    logger.warning("resolve: dropping non-vault-relative path %r", p)
    continue
```

`.startswith("wiki/")` is a prefix match. A path like `wiki/../etc/passwd` passes the check but escapes the vault on filesystem resolution. Same class as my PR #9 N1 (LOOP-GOLD-READER-1 path-traversal defense).

**In production this can't happen** — `target_vault_path` values come from Step 6 `finalize()` which Pydantic-validates the shape. But Step 2's read surface is critical cross-step infra; defense-in-depth is cheap.

**Fix.** One guard:
```python
if ".." in p or p.startswith(".") or "\\" in p:
    logger.warning("resolve: dropping path with traversal/escape characters %r", p)
    continue
```

Or use a stricter regex match against the expected pattern (`^wiki/[a-z0-9-]+/\d{4}-\d{2}-\d{2}_[\w-]+\.md$`). Defer to (a) — simpler.

### 2. Email In-Reply-To graph walk — **no walk; single-query set-membership** ✓

B1 made a design simplification I hadn't anticipated in the task brief: there is NO recursive graph walk. The email resolver performs a **single SQL query** using set-membership on the CURRENT signal's (in_reply_to + references) IDs:

```sql
WHERE source = 'email' AND primary_matter = %s
  AND ( (payload->>'email_message_id') = ANY(%s)
     OR (payload->>'in_reply_to') = ANY(%s)
     OR payload->'references' ?| %s )
ORDER BY created_at DESC LIMIT 3
```

- **Cycle detection:** moot. No recursion; single query. Can't cycle.
- **Depth cap:** irrelevant. `_MAX_PATHS=3` caps the returned set via SQL LIMIT.
- **Equivalence to a recursive walker:** for the common case (replies in an existing thread), the CURRENT signal's `references[]` already contains all ancestors of the thread — walking from the current node is redundant because RFC 5322 says `References:` enumerates the full chain. Set-membership against the current signal's references captures everything a bounded walk would.

**This is cleaner than a walker.** No stack-depth concerns, no cycle-detection state, no recursion. Single SQL query with two-sided membership (`theirs ∈ mine OR mine ∈ theirs` via the `?|` jsonb-array overlap operator).

**Subject-match fallback** is also a single query with `lower(regexp_replace(payload->>'subject', '^(re|fwd|fw|aw|vs)\\s*:\\s*', '', 'gi')) = %s`. No loop. ✓

**Minor observation:** the subject-match normalization happens in BOTH Python (`_normalize_subject`) AND SQL (`regexp_replace`). Python normalizes the CURRENT signal's subject; SQL normalizes PRIOR signals' subjects. Double normalization is fine (and necessary — can't index `regexp_replace(subject)` efficiently without a functional index). Consistent stripping pattern across both. ✓

## 8. Pattern observations

Clean split:
- `kbl/steps/step2_resolve.py` — dispatcher + state machine + DB writes

### Fail-soft philosophy

1. **Voyage transport failure** → `VoyageUnavailableError` → degraded-mode (empty paths + success=False cost row)
2. **Empty vault / no stored embeddings** → short-circuit before Voyage call (no cost)
3. **Resolver-internal exception** → `ResolverError` → `resolve_failed` terminal

Layer 3 is the only path that marks signal failed; layers 1 and 2 continue the pipeline as new arc. Matches §4.3 contract exactly. ✓

### Test-injection seam

`embed_fn` parameter on transcript + scan resolvers is a clean dependency injection for unit tests. Default is a lazy import of `kbl.voyage_client.embed`. No module-level patching needed — tests pass a mock directly. ✓
