### S2 — State-leak on parse failure: signal stuck in `triage_running`

**Location:** [`kbl/steps/step1_triage.py:462-490`](../../kbl/steps/step1_triage.py).

```python
def triage(signal_id, conn, ...) -> TriageResult:
    signal_text = _fetch_signal(conn, signal_id)
    _mark_running(conn, signal_id)  # <-- sets status='triage_running'
    # ...
    try:
        result = parse_gemma_response(raw_text or "")
    except TriageParseError:
        _write_cost_ledger(..., success=False)  # <-- writes cost row
        raise                                   # <-- signal still in 'triage_running'
    # ...
    _write_triage_result(conn, signal_id, result, next_state)  # <-- advance only on success
```

**The problem.** `_mark_running` flips status to `'triage_running'` BEFORE the risky parse. If parse fails, `_write_cost_ledger` writes an audit row and then `raise` bubbles `TriageParseError` to the caller. The signal's status stays at `'triage_running'` — **never rolled back**. If the caller (pipeline_tick, separate ticket, not yet shipped) doesn't explicitly catch and update, the signal is **stuck indefinitely**.

**Inconsistency with peer step impls:**
- **PR #10 (Step 2):** On `ResolverError`, explicitly writes `_write_result(conn, signal_id, (), _STATE_FAILED)` BEFORE re-raising. Status flips to `'resolve_failed'` — operator sees it.
- **PR #11 (Step 3):** On parse failure (retries exhausted), writes empty stub + advances to `awaiting_classify` + `success=False` ledger. Signal flows; no stuck state.
- **PR #8 (Step 1):** On parse failure, writes cost row but **does not update status**. Signal stays in `'triage_running'`.

**Verified by test `test_triage_parse_error_writes_failure_ledger_row_and_raises`** (line 500):
- Cost row written with `success=False` ✓
- Result UPDATE NOT fired ✓
- **Status update NOT asserted either way** — the test confirms current behavior but doesn't catch the stuck-state risk.

**Brief §7 row 3** says: *"Triage | Gemma JSON unparseable | TriageParseError | WARN | R3 retry (pared prompt); 2 failures → inbox | Y after R3"*. PR #8 implements **zero** of this (no internal retry, no pared prompt, no inbox route). Its docstring defers all of that to the caller: *"Caller writes a stub + routes to inbox per §3."* But until the pipeline_tick caller is written with this logic, signals leak.

**Three fix options:**

- **(a) Status rollback** — simplest. In the `except` block, call `_write_failed_state(conn, signal_id)` (sets status to `'triage_failed'` — a terminal state) BEFORE `raise`. Signal is visibly failed rather than silently stuck. Matches PR #10's `ResolverError` pattern.

- **(b) Absorb retry + stub** — match PR #11's pattern. Internal `_RETRY_BUDGET=1`; on retries-exhausted, write stub TriageResult + advance to `awaiting_inbox_route` + cost row `success=False`. No raise. Matches PR #11's behavior and §7 row 3's "2 failures → inbox" directive. Pipeline flows.

- **(c) Status rollback AND caller-retry contract** — write status back to `'awaiting_triage'` on parse failure so the next pipeline tick can re-claim and re-try. Loses deterministic "exactly once per signal" guarantee; caller has to bound total retries.

**Lean (b).** It matches §7 row 3 directly, matches PR #11's shape for cross-step consistency, and doesn't rely on a caller that hasn't been written yet. PR #8 already imports `TriageResult` with default factories — the stub is `TriageResult(primary_matter=None)`.

Either (a) or (b) will unblock the state-leak. (c) is the least robust — a retry loop that never converges stalls the pipeline.

**Paired fix:** the test `test_triage_parse_error_writes_failure_ledger_row_and_raises` should explicitly assert the post-failure status (either `'triage_failed'` terminal OR `'awaiting_inbox_route'` + stub UPDATE). Current test is silent on it — exactly the gap that lets the leak through.

**Note:** whatever status value is chosen (`'triage_failed'` or `'awaiting_inbox_route'`), it inherits S1 — not in KBL-A CHECK. Both S1 and S2 remediations converge on the same migration; but they're independent design choices.

## 8. Pattern observations

### Inv 3 fallback belt-and-suspenders

```python
hot_md_content = load_hot_md()
hot_md_block = hot_md_content if hot_md_content else _HOT_MD_FALLBACK

ledger_rows = load_recent_feedback(conn, limit=20)
rendered = render_ledger(ledger_rows)
feedback_ledger_block = rendered if rendered.strip() else _LEDGER_FALLBACK
```

The second guard (`rendered.strip() else _LEDGER_FALLBACK`) is defensive — `render_ledger` already returns a placeholder string on empty input, but the extra check catches a renderer bug that might leak empty string into the prompt. Good hygiene.

### `_coerce_int_score` explicit bool rejection

```python
if isinstance(value, bool):  # bool is subclass of int
    raise TriageParseError(f"triage_score is a bool: {value!r}")
if isinstance(value, int):
    return value
```

Same pattern as PR #11's `_clean_money`. Correct Python-gotcha handling. ✓

### Score/confidence clamping post-coercion

```python
triage_score = max(0, min(100, _coerce_int_score(data["triage_score"])))
triage_confidence = max(0.0, min(1.0, _coerce_confidence(data["triage_confidence"])))
```

Coerce first (raises if uncoerceable), then clamp. Defensive against Gemma emitting `triage_score=150` or `triage_confidence=1.5`. ✓ Tests `test_parse_score_out_of_range_clamps` + `test_parse_confidence_out_of_range_clamps` verify.

### Glossary is deterministic

```python
def _build_glossary() -> str:
    slugs = sorted(slug_registry.active_slugs())
    ...
```

`sorted(...)` ensures glossary text is stable across calls with identical registry state. Smart — Ollama's cache keys on prompt text; deterministic glossary preserves cache hits.
