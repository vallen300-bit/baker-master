Defer — requires all step impls on main first.

### N2 — Migration name is operator-facing only; no framework to record applied state

**Location:** `migrations/20260418_expand_signal_queue_status_check.sql` header (line 28):

> *"Apply order: manual operator run OR auto-assertion via app-boot path."*

Two application paths:
1. Manual `psql -c "BEGIN; \i migrations/...sql; COMMIT;"`
2. App boot via `_ensure_signal_queue_additions()`

No migration framework (flyway/alembic/manual-tracking table) records which migrations have been applied. The app-boot path runs on every Render restart — fine for idempotent DROP+ADD migrations like this one, but brittle pattern in general.

**Pre-flag.** When the number of migration files crosses ~10, introduce a lightweight tracking table (`kbl_applied_migrations(migration_name TEXT PRIMARY KEY, applied_at TIMESTAMPTZ)`) and a simple runner. Not blocking for PR #12.

### N3 — DOWN section is commented out — cannot be `psql -f`-executed directly

**Location:** `migrations/20260418_expand_signal_queue_status_check.sql:89-109`.

The DOWN section ships with `-- ` comment leaders on every executable line. Operator must strip comments manually OR run the test harness's `_strip_comment_leader` helper.

**Safety tradeoff:** commenting DOWN prevents accidental `psql -f migration.sql` from running both UP and DOWN (net no-op). But in a real disaster-recovery scenario, the operator-UX cost is: must manually de-comment before running.

The `test_migration_down_reverts_to_kbl_a_eight` test uses the `_strip_comment_leader` helper to unwrap the DOWN section for parsing assertion — but that helper isn't in any operator-facing tool.

**Recommend.** Add a comment in the migration header explaining the strip-to-run pattern, OR ship a trivial `scripts/migrations_down.sh` that does the comment-stripping for disaster-recovery. Defer; low priority.

### N4 — Live-PG test creates `signal_queue` rows then rolls back via SAVEPOINT — but outer tx commit at line 309 could strand rows if any earlier assertion fails mid-loop

**Location:** `tests/test_status_check_expand_migration.py:278-309`.

The loop at line 278-295 inserts + ROLLBACK-TO-SAVEPOINT for each of the 34 legal values. If the loop iterates cleanly, no rows persist (each row rolled back via SAVEPOINT). But if an assertion inside the loop raises before the SAVEPOINT rollback, the row persists in the outer transaction.

Actually — looking more carefully — the `try`/`except` structure within the loop (lines 279-295) does `ROLLBACK TO SAVEPOINT sp_ok` + `RELEASE SAVEPOINT sp_ok` in BOTH success and exception paths. So a failure mid-loop rolls back the current SAVEPOINT, then re-raises. The outer `finally` at line 326-334 runs the UP section again on cleanup.

Net: even on failure, the last-known-good-tx state ends with an applied UP and (potentially) one stranded test row. Test DB pollution is minimal — `source='migration_test'` is searchable for cleanup.

**Recommend** (optional): have the outer `finally` also `DELETE FROM signal_queue WHERE source = 'migration_test'` before re-applying UP. Closes the pollution loop. ~2 lines.

Low priority. Assumes a throwaway `TEST_DATABASE_URL`.
