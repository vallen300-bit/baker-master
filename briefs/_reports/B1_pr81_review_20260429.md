## §K — Non-blocking observations

1. **Synchronous `_post_response_update` "Processing…" call BEFORE the `return` (deviation from checklist §E.2).** Lines 342-350 of `triggers/slack_interactivity.py` invoke `_post_response_update` (a synchronous `urllib.request.urlopen` with `timeout=5`) inside the async `slack_interactive` endpoint *before* `return JSONResponse({}, status_code=200)` on line 352. The brief checklist §E.2 explicitly requires the Processing… post be invoked from the BackgroundTask, not from the request handler itself.

   Why this is **not** a STOP-criterion violation:
   - The Phase 5 handlers (the heavy work) ARE properly backgrounded via `BackgroundTasks.add_task` on lines 333-339 — those handlers run AFTER the 200 is sent to Slack.
   - `_post_response_update` is wrapped in try/except (lines 124-135) and capped at `timeout=5`.
   - Slack's response_url is normally <500ms; the deviation is a *theoretical* edge case (Slack API slow), not a runtime failure mode under normal conditions.
   - Tests 1+2 (happy paths) pass cleanly with `_post_response_update` patched out — the endpoint does not depend on it for correctness.

   Why it is still **worth flagging** for a follow-up cleanup:
   - `urllib.request.urlopen` is BLOCKING; in an async function it blocks the event loop for the duration. Under load this hurts other concurrent requests, not just this one.
   - The brief checklist is explicit; complying matches future linting/audit expectations.
   - Trivial fix: schedule it as its own `add_task(_post_response_update, response_url, {...})`, or move the Processing… post into `_run_handler`'s entry block (one line move).

   **Recommended follow-up: open a small follow-up PR to background the Processing… post. NOT a merge blocker.**

2. **`_run_handler` lazy-imports Phase 5 handlers inside the try (line 196).** This is a deliberate boundary so a phase5_act import failure during deploy doesn't take down the slack interactivity router on import. Defensive — note-only.

3. **Test 4 (`test_stale_timestamp`) doesn't assert `_post_response_update.assert_not_called()`.** The 403-response path returns BEFORE reaching the optimistic Processing… post, so the post does not fire. The test asserts `h.assert_not_awaited()` (handler not invoked) which is the load-bearing check; the missing assertion on `_post_response_update` is a small completeness gap that doesn't change the test's semantic coverage. Note-only.

4. **`config.slack.signing_secret` indirection.** `_verify_signature` reads the secret via `config.slack.signing_secret` rather than `os.environ.get("SLACK_SIGNING_SECRET")` directly. This is consistent with existing patterns (`slack_events.py`) and centralises secret loading; correct choice. Note-only confirmation.
