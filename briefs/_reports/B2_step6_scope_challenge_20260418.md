## Per-job analysis

### Job 1 — Metadata (source IDs, timestamps, `author: pipeline`)

**Inputs available pre-Step-6:**
- `signal_queue.id` — signal_id
- `signal_queue.source` — source enum
- `signal_queue.created_at` — arrival timestamp

**Deterministic-feasible:** Yes. Every required value is either already on `signal_queue` (typed) or in `payload` JSONB (typed). Frontmatter assembly is `yaml.safe_dump(metadata_dict)`.

**LLM-required edge cases:** None. There is no scenario where "what's the source message_id" requires reasoning — it's a database lookup. Even ambiguous-source signals (e.g., a forwarded email containing a quoted WA message) have one canonical source per `signal_queue` row (the row that the sentinel inserted).

**Recommendation:** **Move to code.** ~10-line `_build_frontmatter(signal_row)` function. No model.

### Job 2 — `related_matters[]` cross-links

**The critical insight:** the cross-link decision is made at **Step 1** (triage prompt outputs `related_matters: [...]`), not at Step 6. Step 4's decision table (§4.5) routes signals with non-empty `related_matters` to "full_synthesis + cross-link flag." By the time Step 6 fires:

- `signal_queue.related_matters` is already a populated JSONB array
- `step_5_decision` already says whether to cross-link

Step 6's job is to **write the cross-link**, not infer it.

**The write is templated:**
```markdown
<!-- generated by KBL pipeline -->
- [{date}] {signal_title} → see [{other_matter}/{entry}.md]({other_matter}/{entry}.md)
```

Append-only, idempotent (dedupe by source signal_id), zero judgment.

**Edge case from task brief: "a signal about 'Cupials defects' should also cross-link to 'Hagenauer RG7' — does that need LLM semantic reasoning?"**

If `related_matters[]` from Step 1 already includes `hagenauer-rg7`, no — it's deterministic. If `related_matters[]` does NOT include it but should, that's a **Step 1 prompt quality issue**, not a Step 6 design issue. Don't paper over upstream gaps with downstream LLM calls — fix the upstream prompt (Step 1's disambiguation block already handles this for the brisen-lp ↔ hagenauer-rg7 case; same shape works for cupial ↔ hagenauer-rg7).

**A different system would be:** a co-occurrence graph built from prior vault entries, suggesting cross-links beyond what Step 1 sees. That's interesting but **out of KBL-B scope** — it's a "vault intelligence" feature for KBL-C or later. Don't smuggle it into Step 6.

**Deterministic-feasible:** Yes. Iterate `related_matters`, format markdown line, append to `wiki/<m>/_links.md` per entry. ~20-line implementation.

**LLM-required edge cases:** None within KBL-B scope.

**Recommendation:** **Move to code.**

### Job 3 — Frontmatter schema validation

**Inputs:** Opus's `opus_draft_markdown`, parsed into frontmatter dict + body string.

**Validation:** Pydantic model with `Literal["opportunity", "threat", "routine"]` for vedana, `Literal[<active_slugs>]` for primary_matter, etc. Catches:
- Wrong types (string instead of list for related_matters)
- Missing required keys
- Extra keys not in schema (set `model_config = ConfigDict(extra="forbid")`)
- Out-of-enum values

**Failure mode A — Pydantic-malformed:** Opus produced unparseable YAML or wrong-shape values.
- Brief's current proposal: Sonnet repair pass.
- My counter: **Opus R3 retry with pared prompt** ("you returned X, expected Y; please fix and re-emit").
- Why Opus over Sonnet for repair:
  - Opus already has full context (signal + thread + entities) loaded; Sonnet would need the same context to repair correctly → token-cost difference shrinks
  - Opus 4.7 is more capable than Sonnet 4.6 — first-try repair success is higher
  - One model dependency, not two
  - Same retry ladder as Step 5's existing R3 (no new error-handling shape)

**Failure mode B — Pydantic-valid-but-stylistically-off:** vedana value is in the enum, primary_matter is canonical, but the frontmatter "feels off" (e.g., `triage_score: 87.3` when integers are expected, or `related_matters: ["Hagenauer RG7"]` instead of `["hagenauer-rg7"]`).

These are **table-driven canonicalization**, not LLM reasoning. `int(round(x))`, `slug_registry.normalize(s)`. Already wired post-Step-1.

**LLM-required edge cases:** None where Sonnet adds value over Opus retry. The only argument FOR Sonnet was "cheaper than Opus" — but if repair happens <5% of signals (Opus 4.7 with explicit schema in system prompt is reliable), the per-day cost delta is sub-$0.50 vs the ongoing complexity cost of maintaining a second model + prompt.

**Recommendation:** **Move to Pydantic. Repair via Opus R3, not Sonnet.**

### Job 4 — Tone/style polish on prose body

**Is this even necessary?** The brief offers **zero evidence** that Opus 4.7 drafts need stylistic polish.

What's the evidence base?
- Cargo-cult pattern: "draft + refine" with two model passes was a 2023 GPT-3.5/4 idiom
- Opus 4.7 at 1M context with a clear system prompt produces vault-quality prose first try
- Vault entries are internal knowledge artifacts, not customer-facing copy — polish bar is "Director can re-read it without wincing," not "publishable"
- The reviewer is the same person who wrote the system prompt that produced the draft (Director, indirectly via AI Head); the writing voice IS the desired voice

**The cargo-cult test:** if we delete Job 4 and ship Phase 1 with no polish pass, what breaks?
- Wiki entries land slightly less polished
- Director re-reads them in `wiki/_inbox/<…>` review
- If they're consistently ungainly: add Sonnet polish in v2 (cheap to add later)
- If they're fine: we never paid the latency / cost / dependency / failure-mode tax

**Asymmetric bet:** the cost of NOT having polish in v1 is bounded (worst case: a few rough wiki entries Director can lightly edit). The cost of HAVING polish in v1 when it's not needed is recurring (Sonnet API dependency, retry ladder, schema state, prompt to maintain, every signal pays the latency).

**Recommendation:** **Delete this job.** Re-add only if Phase 1 burn-in produces concrete evidence that Opus drafts are deficient. Set explicit success criterion in §13: "if Director marks <5% of wiki entries as 'needs editing' across 100-signal Phase 1, no polish pass needed."

## Gemma-for-Step-6 alternative

**Setup:** keep one Step 6 LLM call but use Gemma 4 8B (local, free, fast) instead of Sonnet.

**What breaks:**
- Gemma at 76m matter / 88v vedana is below threshold for *semantic* decisions (which is why Layer 2 ALLOWED_MATTERS gate exists at Step 5 entry per D3). Step 6's matter-touching work (Job 2 cross-links) IS downstream of Step 1 + Layer 2 — so the semantic safety net is already applied.
- Gemma 4 8B prose polish risk: hallucination during rewrite. Polish is a paraphrase task, and Gemma at 8B is known to drop facts mid-paraphrase (vs Opus/Sonnet which preserve verbatim quotes faithfully). Rewriting a wiki entry that says `"€2.5M shareholder loan"` and getting back `"€2 million shareholder loan"` is silently corrupting Director's knowledge base.
- Latency: Gemma 6-15s/call (per Step 1 estimate) on macmini. Polish + cross-link assembly per signal adds ~10-20s end-to-end. Acceptable but unnecessary if we have nothing for Gemma to do.

**Residual risk specific to Gemma at Step 6:** fact-corruption-during-paraphrase. Sonnet 4.6 has materially lower rate; Opus 4.7 essentially zero rate.

**Verdict:** Gemma-for-Step-6 is **not a good middle ground**. If we keep an LLM call, Sonnet is the right model. But the prior analysis says we shouldn't keep an LLM call at all.

## Recommended Step 6 shape (concrete spec for §4.7 rewrite)

**Step 6 — `finalize` (deterministic finalization)**

**Reads:** `opus_draft_markdown`, `primary_matter`, `related_matters`, `extracted_entities`, `source`, `signal_id`, `created_at`, `payload` (for source-specific IDs), `step_5_decision`.

**Writes:** `final_markdown TEXT`, `target_vault_path TEXT`.

**Side effects:** Append cross-link stub to `wiki/<m>/_links.md` for each `m in related_matters`. Idempotent (dedupe by source signal_id within the file).

**Ledger:** No row (no model call).

**Log:** On Pydantic validation failure only (`level='WARN'`, `component='finalize'`, `message=<failed key + reason>`). On cross-link write failure only (`level='ERROR'`, `component='finalize'`, `message='cross-link write failed: <path>'`).

**Mechanism:**

```python
def step6_finalize(signal: SignalRow) -> tuple[str, str]:
    """Returns (final_markdown, target_vault_path).

    Raises FinalizationError on Pydantic validation failure → upstream
    catches and triggers Opus R3 retry on Step 5.
    """
    frontmatter_dict, body = _split_opus_draft(signal.opus_draft_markdown)

    # Augment with deterministic metadata
    frontmatter_dict.update({
        "signal_id": signal.id,
        "source": signal.source,
        "primary_matter": signal.primary_matter,
        "related_matters": signal.related_matters,
        "vedana": signal.vedana,
        "triage_score": int(signal.triage_score),
        "triage_confidence": float(signal.triage_confidence),
        "created_at": signal.created_at.isoformat(),
        "source_id": _extract_source_id(signal),  # message_id / chat_id+msg_id / meeting_id
        "author": "pipeline",
    })

    # Pydantic validate (raises if malformed)
    validated = WikiFrontmatter.model_validate(frontmatter_dict)

    # Compute path — convention: wiki/<matter>/<YYYY-MM-DD>_<slug-of-summary>.md
    target_path = _compute_vault_path(signal, validated)

    # Reassemble final_markdown
    final = "---\n" + yaml.safe_dump(validated.model_dump(), sort_keys=False) + "---\n\n" + body.strip() + "\n"

    # Cross-link writes (idempotent per signal_id)
    for related in signal.related_matters:
        _append_cross_link_stub(related, signal, target_path)

    return final, target_path
```

**Failure modes:**

| Failure | Detection | Recovery |
|---|---|---|
| `_split_opus_draft` raises (no frontmatter delimiter, malformed YAML) | Exception | Trigger Step 5 R3 retry with pared prompt: "your previous output had no '---' frontmatter delimiter — re-emit with strict frontmatter + body separation". |
| Pydantic validation fails (wrong type, missing key, out-of-enum value) | `ValidationError` | Trigger Step 5 R3 retry with pared prompt naming the failed field. |
| Cross-link `_links.md` write fails (filesystem permission, lock contention) | OSError | Log ERROR, set state to `finalize_failed`, do NOT advance to Step 7. Manual review required. |
| Both R3 retries on Step 5 still produce malformed draft | After max retries | Set `state='failed'`, route to inbox with full Opus output preserved in `opus_draft_markdown` for Director review. |

**Latency:** ~50-200ms (file I/O + serialization + cross-link append). Fastest step in the pipeline by 2 orders of magnitude.

**Invariant (§4.7 restated):** `final_markdown` is YAML-frontmatter + Markdown body, parseable. `target_vault_path` starts with `wiki/` and ends `.md`. No partial writes — Pydantic-fail leaves both as NULL (signal goes back to Step 5 for retry).
