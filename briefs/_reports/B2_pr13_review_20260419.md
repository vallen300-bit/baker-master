## 3. Should-fix

### S1 — Missing forced-emit test for `CROSS_LINK_ONLY` runtime guard

**Location:** `tests/test_step4_classify.py` — no test exists.

The classifier has a defense-in-depth runtime check (`kbl/steps/step4_classify.py`):

```python
if decision is ClassifyDecision.CROSS_LINK_ONLY:
    _mark_failed(conn, signal_id)
    raise ClassifyError(
        "CROSS_LINK_ONLY is a Phase 2 reserved decision; no Phase 1 "
        "rule maps to it. Evaluator emitted it — check decision table."
    )
```

Brief §Specific-Scrutiny item 2 enumerates three sub-checks:

1. ✓ Guard exists (raise on emit, not silent fallback).
2. ✓ No rule path can reach it (verified by code inspection — all 5 `return` statements in `_evaluate_rules` yield `SKIP_INBOX` / `STUB_ONLY` / `FULL_SYNTHESIS`; only the `raise ClassifyError` Rule-0 path is the other exit).
3. ✗ **Test exists that asserts the guard fires on forced attempt** — missing.

The only `CROSS_LINK_ONLY` reference in tests is `test_classify_decision_enum_values_stable` asserting `.value == "cross_link_only"` — purely an enum-shape check, not a guard-fire assertion.

**Why this matters.** The guard is defensive code against a future refactor that accidentally maps a rule to `CROSS_LINK_ONLY`. Without a test, any such refactor produces a silent regression in test coverage — the guard becomes untested dead code that could be unintentionally broken (e.g., someone removing the `is` check during a type-annotation tidy-up) without any test catching it.

**Fix.** Add a ~12-line test. Sketch:

```python
def test_classify_rejects_cross_link_only_from_evaluator() -> None:
    """Runtime guard: if the evaluator ever emits CROSS_LINK_ONLY
    (Phase 2 reserved), classify() must mark the signal failed and
    raise ClassifyError — never write cross_link_only to the DB."""
    conn = _classify_conn(triage_score=80, primary_matter="movie")
    with _patch_scope("movie"), patch.object(
        step4_classify,
        "_evaluate_rules",
        return_value=(ClassifyDecision.CROSS_LINK_ONLY, False),
    ):
        with pytest.raises(ClassifyError, match="Phase 2 reserved"):
            classify(signal_id=77, conn=conn)

    failed_rows = [
        c for c in conn._calls
        if "update signal_queue set status = %s" in c[0].lower()
        and c[1] == ("classify_failed", 77)
    ]
    assert failed_rows
    decision_rows = [
        c for c in conn._calls
        if "step_5_decision" in c[0].lower()
    ]
    assert not decision_rows  # no decision write on guard
```

Trivial add. After this, REDIRECT → APPROVE automatic.

## 4. Nice-to-have

### N1 — Unknown-slug pass-through in `KBL_MATTER_SCOPE_ALLOWED` not documented

Brief §Specific-Scrutiny item 5 asks: *"Invalid slugs (not in `slugs.yml` v9) — does parser reject, warn, or pass-through? Flag the chosen behavior."*

**Chosen behavior: silent pass-through.** `_get_scope_env_override()` splits/strips/dedups tokens with zero reference to `slugs.yml`. A typo like `KBL_MATTER_SCOPE_ALLOWED=cupila` (missing `l`) silently enters `allowed_scope`, never matches any canonical `primary_matter`, and effectively no-ops — a signal with `primary_matter='cupial'` would NOT gain scope via the typo.

**Safety assessment.** Pass-through is safe (the misspelled slug can't match a canonical primary_matter), but silently ignoring operator error is a soft trap. Two small ergonomic improvements would help:

1. A one-line docstring note in `_get_scope_env_override()`: *"Tokens are not validated against `slugs.yml`; unknown slugs pass through and simply never match."*
2. An optional WARN log at load time if a token doesn't match any canonical slug — off by default, opt-in via a verbose flag if you want to avoid startup log noise.

Neither is blocking; document behavior first, automate later.

### 5.8 Failure-path atomicity (item 8)

```python
try:
    decision, cross_link_hint = _evaluate_rules(...)
except ClassifyError:
    _mark_failed(conn, signal_id)
    raise
```

`_mark_failed` executes the UPDATE before `raise` re-throws. Explicit test `test_classify_below_threshold_raises_and_marks_failed` asserts both:
1. A status UPDATE to `classify_failed` exists in `conn._calls`
2. No `step_5_decision` UPDATE fires on the failure path

**Caveat — caller-owns-commit semantic.** `classify()` has zero `conn.commit()` calls. Same pattern as Step 1 / Step 2 / Step 3 (`grep -nE 'commit\b' kbl/steps/step[123]_*.py` returns zero matches). If the caller rolls back the connection on exception, the `classify_failed` UPDATE is lost and the row stays at `awaiting_classify` — the signal would be re-claimed on next tick and hit the same ClassifyError forever. This is a **caller contract**, not a Step 4 drift: Step 4 matches Steps 1-3 precedent exactly. Not flagged; the pipeline tick harness (not yet in this PR scope) owns the commit-on-exception vs rollback-on-exception decision.
