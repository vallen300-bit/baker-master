## Should-fix, deferrable (S2)

### S2 — `_increment_retry_count` runs `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` in the hot path

**File:** [kbl/steps/step6_finalize.py:417-436](kbl/steps/step6_finalize.py:417)

```python
def _increment_retry_count(conn: Any, signal_id: int) -> int:
    with conn.cursor() as cur:
        cur.execute(
            "ALTER TABLE signal_queue "
            "ADD COLUMN IF NOT EXISTS finalize_retry_count INT NOT NULL DEFAULT 0"
        )
        cur.execute("UPDATE signal_queue SET finalize_retry_count = ... RETURNING ...")
        ...
```

Every validation-failure path issues a DDL statement. PG's `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` takes an `ACCESS EXCLUSIVE` lock on `signal_queue` even when it's a no-op (PG must verify the column exists before short-circuiting). On the single-writer pipeline today this is harmless, but it's non-idiomatic — DDL belongs in migrations, not application paths.

The module docstring acknowledges the choice: *"the retry counter is not part of the Step 6 migration on purpose (R3 coordination with Step 5 is the reason, see module docstring)"* — but the module docstring itself doesn't actually explain the coordination rationale. I don't see why this column couldn't simply live in `migrations/20260419_step6_signal_queue_final_markdown.sql` alongside `final_markdown` + `target_vault_path`.

**Fix (1-line migration + 1 deletion):**
```sql
ALTER TABLE signal_queue
  ADD COLUMN IF NOT EXISTS finalize_retry_count INT NOT NULL DEFAULT 0;
```
…and drop the inline `cur.execute("ALTER TABLE...")` call. The `COALESCE(finalize_retry_count, 0) + 1` already handles pre-column-exists environments defensively; the column-add belongs beside the migration.

**Not a blocker** because: (a) IF-NOT-EXISTS is a safe no-op in every practical deployment order; (b) concurrency in KBL today is single-tick-per-row (FOR UPDATE SKIP LOCKED); (c) the only cost is AEL acquisition per validation-failure, which is bounded by the retry count (max 3 per signal).

Defer to either a follow-up 1-line migration PR or fold into Step 7's PR.

## Subtle points worth noting (not findings)

1. **Two-level Inv 9 pin is the best structural invariant enforcement I've seen in KBL so far.** Runtime mocks catch intent; source scan catches regression. When a future edit accidentally imports `tempfile` or calls `Path.write_text`, the source scan fails with a specific file + forbidden-pattern message. That's a durable forcing function.

2. **`money_mentioned` lives in `opus_draft_markdown` as raw strings; Step 6 normalizes via `_parse_money_string` before Pydantic sees the dict.** This preserves prompt-cache stability (Opus emits stable raw strings) while giving Silver documents structured money. The dropped-unparseable path is logged only via the Director-visible `opus_draft_markdown` audit trail (the raw string is still there for inspection) — no explicit WARN row for individual money-parse drops. Defensible: a WARN per money entry would be noisy; the aggregate audit path works.

3. **Status provenance gate is the subtle correctness win.** §3.7 prevents Opus from smuggling `status: stub_auto` into a full-synthesis draft (which would confuse Step 7's downstream readers) and also prevents deterministic stub writers from forgetting to set status. Both paths go straight to `finalize_failed` terminal — they're pipeline bugs, not draft issues, so R3 retry doesn't help. Good taste.

4. **`CrossLinkStub.render_stub_row` pipe-escapes `|` in excerpts.** Cheap but important — otherwise a signal with `"€3000 | €5000"` in its title breaks Markdown pipes in `_links.md`. Not brief-required but correct.

5. **`primary_matter=null` + `status=stub_inbox` + `related_matters=[]` is the coherent stub_inbox shape.** R7 enforces the first two; OQ7 clarified the third. All three combine to produce `wiki/_inbox/...` path via the `_inbox` carve-out. Orthogonal enforcement — nice design.

6. **Test 6 (cost-cap pause) in pipeline_tick now checks Step 6 is gated.** Correctly extended the PR #14 test to verify `mocks["step6"].call_count == 0` after the status check sees `paused_cost_cap`. This is the scenario where the cost spigot closes — signal waits for operator reset or midnight-UTC cap rollover.
