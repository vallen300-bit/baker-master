## 5. Scoring bugs — 2 instances fixed, 1 instance remaining

### 5a. Fixed in v3: null-string normalization

`normalize_matter()` now treats `"null"` / `"none"` as Python `None`. Test: all 5 cases pass. No longer costs rows.

### 5b. NOT fixed — stale `MATTER_ALIASES["brisen-lp"]`

Pre-SLUGS-1, `wertheimer` was an alias for `brisen-lp`. SLUGS-1 separated them into distinct canonical slugs. But `scripts/run_kbl_eval.py:MATTER_ALIASES` is the local (non-registry) copy and was NOT updated:

```python
MATTER_ALIASES = {
    ...
    "brisen-lp": ["brisen", "wertheimer"],   # ← stale, wertheimer is now its own slug
    ...
}
```

Consequence: model output `wertheimer` normalizes to `brisen-lp`, comparison against label `wertheimer` fails. 1 Gemma miss, 1 Qwen miss.

**I did not fix this** because the task scope guardrails say "Do NOT re-open the slug set itself. SLUGS-1 owns the canonical list." I interpreted `MATTER_ALIASES` as inside that scope. If it belongs instead with scoring, I should have fixed it. Flagging for AI Head judgment.

**Fix when SLUGS-1 merges:** delete `MATTER_ALIASES` entirely, use `kbl.slug_registry.normalize()`. The registry already reflects SLUGS-1 truth; keeping two alias dicts is the failure mode that produced this bug.
