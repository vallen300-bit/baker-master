## 4. Nice-to-have

### N1 â€” Defense-in-depth path-traversal guard on `matter` slug

**Location:** [`kbl/loop.py`](../../kbl/loop.py) â€” `load_gold_context_by_matter` argument validation.

`matter_dir = vault_root / _WIKI_SUBDIR / matter` uses Python's `Path` join, which does NOT resolve `..` segments. But `Path.exists()` / `.iterdir()` resolve them at the OS level. A malicious or buggy caller passing `matter="../../etc"` would escape the `wiki/` directory.

In production this can't happen â€” `primary_matter` flows from Step 1 (Gemma) â†’ `slug_registry.normalize()` â†’ canonical slug, no path chars. The helper docstring explicitly delegates normalization to the caller for slug-registry decoupling (correct design choice). But Leg 1 critical infra deserves defense-in-depth on the read surface.

**Fix.** One guard at the top of the function:
```python
if "/" in matter or "\\" in matter or matter.startswith(".") or ".." in matter:
    raise LoopReadError(f"matter must be a single path component (got {matter!r})")
```
~3 lines + 1 test. Keeps the slug-registry decoupling intact.

**Why nice-to-have, not should-fix:** the production caller is well-behaved; the worst-case escape lands at most one segment up (`vault_root/<other-dir>/.md`-shaped reads), Pydantic at Step 6 finalize() would reject the result anyway, and the resolved-path mismatch surfaces as a `LoopReadError` if the path doesn't exist. Defense-in-depth is cheap to add later.

### N2 â€” Non-date-prefix filename ordering is silently lexicographic

**Location:** [`kbl/loop.py`](../../kbl/loop.py) â€” `md_files = sorted(...)` line.

Tests verify date-prefix files (`2026-04-01_...`) sort chronologically. âœ“

But the sort is purely lexicographic. A file like `99_archive.md` would sort AFTER `2026-04-05_draft.md` (digit `9` > digit `2` in ASCII). A file like `index.md` sorts even later (`i` > `9`). So a Director-authored `README.md` or `index.md` in a matter dir lands at the END of the chronological reading order, regardless of when it was authored.

This is mildly counter-intuitive and not exercised by tests. For the canonical date-prefix convention it's fine; for mixed naming it loses chronology.

**Fix options:**
- **(a)** Document explicitly that non-date-prefix files lose chronological order. One-sentence docstring addition. Cheapest.
- **(b)** Sort by frontmatter `created` field if present, falling back to filename. More correct but requires re-reading frontmatter for sort. Slower.
- **(c)** Skip non-date-prefix files entirely, with a logger warning. Most opinionated; could surprise Director.

Lean (a). Defer (b) until a real Phase-3 use case warrants the cost.

### N3 â€” UTF-8 BOM in vault files would silently exclude them

**Location:** [`kbl/loop.py`](../../kbl/loop.py) â€” `_read_frontmatter` line `if not content.startswith("---"): return None`.

If a vault file has a UTF-8 BOM (`\ufeff`) before its frontmatter (some editors emit this), the `startswith("---")` check fails, the file is treated as Silver, and silently excluded.

In practice no one emits BOM on Markdown files in 2026; vim/VS Code/Obsidian don't add it by default. But Microsoft tools do (Notepad, some Outlook copy-paste paths). A Director who happens to create a Gold file via Outlook â†’ save would have it silently excluded.

**Fix.** Strip BOM in `_is_gold_file` before the frontmatter check: `content = content.lstrip("\ufeff")`. One line.

Low priority. Defer until observed.

B3's Step 5 prompt loads Gold for `primary_matter` only â€” not for entries in `related_matters`. This is the right scoping decision now (per my N6 in the STEP5-OPUS review). ### N5 â€” Frontmatter-in-each-block token cost at Phase 3 scale

**Location:** Out-of-scope for this PR.

Each Gold block includes the file's full frontmatter verbatim. For Phase 1 (~3-10 entries/matter), the YAML overhead is ~150-300 tokens per matter â€” negligible. For Phase 3 (~200 entries/matter as B3 OQ3 anticipates), YAML overhead alone is ~6K tokens before any body content.
