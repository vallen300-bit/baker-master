## S1 — Inv 4 race (blocking)

### The scenario

1. Director promotes a Silver → Gold on some other machine (laptop, web
   edit, etc.) and pushes to `origin/main`. The commit has `author: director`
   + `voice: gold` in the frontmatter of `wiki/ao/2026-04-19_tonbach.md`.
2. Mac Mini's local vault clone has not yet pulled. Its copy of the file
   either does not exist or still has `author: pipeline`.
3. Pipeline_tick dequeues a signal where Step 6 produced
   `target_vault_path = wiki/ao/2026-04-19_tonbach.md` (same matter,
   same day, same title slug — realistic collision).
4. `commit()` runs:
   - line 574: `_mark_running` — claim + internal commit. OK.
   - line 577: `main_abs = (vault / target_vault_path).resolve()`.
   - line 578: `_assert_path_inside_vault` — OK.
   - **line 579: `_inv4_guard_target_path(main_abs)`** — reads **local** FS.
     File either does not exist or does not yet have `author: director`.
     Guard returns OK.
5. line 588: `acquire_vault_lock`. OK.
6. line 589: `_git_pull_rebase(cfg)` — fast-forwards Director's commit
   into the working tree. `wiki/ao/2026-04-19_tonbach.md` now has
   `author: director` on local FS.
7. line 592: `_atomic_write(main_abs, row.final_markdown)` — writes the
   pipeline's Silver frontmatter (`author: pipeline`) on top.
   **Director's Gold is gone.**
8. `git add` + `git commit` + push. Remote now has the pipeline
   overwrite. Gold is lost unless recoverable from reflog.

### Why the current test doesn't catch this

`test_commit_inv4_collision_refuses` (lines 566-600 of `test_step7_commit.py`)
seeds the Director-authored file on the **local** vault clone first,
then pushes. At `commit()` entry the file is visibly on local FS, so
line 579's guard fires. The test passes, but it never exercises the
remote-only case.

### Fix (minimal diff)

```python
# commit() around line 587 — move the Inv 4 guard inside the flock,
# after the pull.

try:
    with acquire_vault_lock(cfg.lock_path, cfg.flock_timeout):
        _git_pull_rebase(cfg)

        # Inv 4 AFTER pull: any Director promotion pushed since our
        # last sync is now in the working tree.
        _inv4_guard_target_path(main_abs)

        try:
            _atomic_write(main_abs, row.final_markdown)
            ...
```

And delete line 579 (the pre-flock call). `_assert_path_inside_vault`
can stay outside the flock (path containment is purely lexical, no FS
dependence).

## S2 — deferrable

### S2.1 — `git pull --rebase` leaves rebase state on file-level conflict

`_git_pull_rebase` raises `CommitError` when rebase fails. The caller
(`commit()`) catches `CommitError` and calls `_git_hard_reset_one` →
`git reset --hard ORIG_HEAD`. But `git reset --hard` during an ongoing
rebase (i.e. `.git/rebase-merge/` exists) does **not** abort the rebase —
it just moves HEAD. Next tick's `git pull --rebase` will complain about
the in-progress rebase.

Fix: before `_git_hard_reset_one`, call `git rebase --abort` best-effort
(with its own `try/except subprocess.CalledProcessError: pass`). Or do
a pre-flight `git rev-parse --is-inside-rebase` check.

Low real-world risk because Inv 9 single-writer makes mid-rebase
conflicts rare (only happens if Director pushes a file-level conflict
against a pipeline-authored file at the same path between our last
pull and this push). But the recovery path requires manual operator
intervention, which is worth avoiding in production.

### S2.2 — `_git_add_commit` raises on no-op diff

If a retried `commit()` writes identical content to what's already at
HEAD (e.g. a crash after git-commit but before DB-commit left the
working tree clean post-pull), `git status --porcelain` returns empty
and `_git_add_commit` raises `CommitError("git status reports no
changes to commit")`. Signal goes to `commit_failed` + operator
investigates — sane failure mode but creates a false-positive that
looks like a real commit failure in metrics.

Consider: if `git status --porcelain` is empty, treat it as a successful
no-op — skip the commit, capture `HEAD` as `commit_sha`, proceed to DB
finalize + TOAST cleanup. Makes the retry path idempotent.

Deferrable — rare edge case, current behaviour is correct if loud.

## N — nits (non-blocking)

**N1.** `_fetch_unrealized_stubs` SQL orders by `target_slug ASC`. The
file sort (DESC by date) is a separate concern inside
`_append_or_replace_stub`. Comment on the DB `ORDER BY` clarifying that
the in-file sort is independent would save a reader a few seconds.

**N2.** `_extract_primary_and_title` regex `_TITLE_RE` strips surrounding
whitespace but not surrounding quotes. If Step 6 ever emits `title: "Foo
bar"` (Pydantic may insert quotes on certain characters), the commit
message becomes `Silver: ao — "Foo bar" (sig:10)` with literal quotes.
Step 6's current YAML emission avoids this, so it works today, but a
regex tweak (`strip('"\' ')` on the captured group) would be robust.

**N3.** `_DIRECTOR_AUTHOR_RE = re.compile(r"^author:\s*director\s*$",
re.MULTILINE)` does not match `author: "director"` or `author: director
# promoted 2026-04-19`. Silver's Pydantic output is plain, so current
guard is tight enough, but a Director who hand-edits with quotes or a
trailing comment would slip through. Consider `r"^author:\s*\"?director\"?\s*(#.*)?$"`.

**N4.** `_git_checkout_discard` calls `git clean -fd` which would remove
any untracked files in the vault, including scratch files the operator
may have intentionally placed. Low likelihood given Inv 9 single-writer,
but `-f` is a big hammer. Consider narrowing to `git clean -fd -- wiki/`
or `git clean -fd -- <rel_paths>` — only remove untracked where we
touched.

**N5.** `_mark_completed` has inline `ALTER TABLE … ADD COLUMN IF NOT
EXISTS committed_at / commit_sha`. Same pattern I flagged in PR #15 for
`finalize_retry_count`. Once the follow-on migration lands (the
`20260419_step7_signal_queue_commit_meta.sql` or similar that adds
`committed_at`/`commit_sha` to signal_queue), these inline ALTERs should
be deleted — they defend against first-run schema drift but they
execute on every signal's terminal flip otherwise (a tiny but repeated
cost).

**N6.** Lock file at `{vault}/.lock`. If the operator's vault
`.gitignore` doesn't include `.lock`, the lock file itself becomes
git-tracked, polluting history. Small operational risk. Consider adding
`.lock` to the `.gitignore` on vault bootstrap, or use a name under a
known-ignored dir (e.g. `.git/_vault.lock` — still flock-able).

## What's excellent

- **Flock helper (`kbl/_flock.py`)** — 70 lines, single-responsibility,
  uses `O_RDWR | O_CREAT` to avoid unlink-create races, re-uses the
  same lock file forever (no stale-lock cleanup needed because `flock`
  drops on close), 100 ms polling with monotonic deadline, releases on
  exception. Textbook.

- **`_atomic_write`** — `tempfile.NamedTemporaryFile(dir=final_path.parent)`
  ensures same-filesystem rename (os.replace atomic on POSIX),
  `os.fsync(tf.fileno())` before close, cleans up the orphan tmp on
  replace failure. Pattern is tight.

- **`_append_or_replace_stub`** — the regex-based marker replacement +
  DESC-date sort + prefix preservation is correctly layered. The
  "non-stub line after a stub row" edge case is handled (pinned at the
  tail). I expected this to be the buggy corner and it isn't.

- **Rebase-retry + reset-hard on exhaustion** — `_git_push_with_retry`
  is precisely shaped: one retry with pull-rebase between, no more, no
  less. The caller's reset-hard-ORIG_HEAD on `CommitError` keeps the
  local clone in sync with remote HEAD — next tick starts from a sane
  place.

- **TOAST cleanup in same UPDATE as terminal state flip** — atomic
  all-or-nothing per brief §4.9 item 15. The `status = 'commit_running'`
  guard on the WHERE clause prevents a concurrent second worker from
  double-TOAST-ing (which shouldn't happen anyway given the internal
  commit on claim, but nice).

- **Mock-mode default-OFF parsing** —
  `os.environ.get(_DISABLE_PUSH_ENV, "").lower() in ("1", "true", ...)`
  correctly defaults to False on unset. `test_commit_mock_mode_does_not_push`
  verifies remote stays at seed commit.

- **CHECK constraint alignment** — `awaiting_commit`, `commit_running`,
  `commit_failed`, `completed` all in the 34-value set from PR #12.
  Verified via grep on `20260418_expand_signal_queue_status_check.sql`
  lines 79-83.

- **7-step happy path test** —
  `test_process_signal_happy_path_commits_once_per_step` locks in
  `commit_count == 7`, `rollback_count == 0`, strict `[step1..step7]`
  ordering. This is the forcing function that survives every future
  refactor.

- **Step 7 terminal-internal-commit-then-rollback test** —
  `test_process_signal_step7_terminal_flip_internal_commit_then_rollback`
  mirrors Steps 1/4/5/6 pattern. Orchestrator sees commit=7, rollback=1.
  Sealed `commit_failed` state survives outer rollback.

- **Real git repos under `tmp_path`** —
  `test_step7_commit.py` uses `git init --bare` + `git clone` for the
  vault fixture. Happy-path runs the real `git pull/add/commit/push`
  subprocess. Fault injection is narrow `subprocess.run` patches only.
  This is the right tradeoff: fast enough (~3s for 26 tests), catches
  real git behaviour, mocks only what must be mocked.
