## Subtle points worth noting (not findings)

1. **Test 6's `commit==6` is semantically overcounting a real PG.** The "orchestrator commit after Step 5 returns" fires on an already-committed transaction — in real PG this is an empty `COMMIT;` on an empty transaction block. Fine. The test correctly counts both commits on the MagicMock. When the impl ever gets wrapped in a `with conn.transaction()` style block, this test will need a minor update, but that's a future concern.

2. **Test 7's sentinel approach is elegant.** Rather than trying to prove a negative by patching, it asserts the *namespace* doesn't yet expose Step 6/7 — which is the exact condition the dispatch brief wanted pinned. When Step 6 ships, this test breaks with a helpful message ("update orchestrator wiring + this test"), acting as a forcing function for the wire-up.

3. **The `test_process_signal_step2_raises_resolver_error_preserves_step1_commit` docstring is careful.** It correctly notes the MagicMock `commit==1` is an artifact of the mock, not a statement about PG semantics — in real PG, Step 1's commit sealed the Step 1 writes, and the Step 2 rollback rolls back only the Step 2 fragment. This nuance was easy to mis-explain; B1 got it right.
