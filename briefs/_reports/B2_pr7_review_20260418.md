## 3. Should-fix

### S1 — Phone normalization gap on `0041` international trunk prefix

**Location:** [`baker/director_identity.py:46-63`](../../baker/director_identity.py).

The `_normalize_phone` helper strips all non-digits via `re.sub(r"\D", "", raw)`. For `+41 79 960 50 92` / `+41799605092` / `41799605092@c.us` this collapses to the canonical `41799605092` and matches `DIRECTOR_PHONES`. ✓

But `0041 79 960 50 92` collapses to `00041799605092` — 14 digits, **does not match** the 11-digit canonical. The docstring acknowledges this:

> *"Note the `0041` case: some WhatsApp ports preserve an `00` trunk prefix, giving a 14-digit result that does NOT match the 11-digit canonical. This is accurate — Swiss trunk-prefix `00` is not the same as the empty plus-replacement. If Director adds a format that serializes as `00…` we add it to DIRECTOR_PHONES explicitly."*

**Why this matters.**

`00` is the **international call prefix** — it's the standard alternate to `+` for E.164 numbers in Europe (and most of the world outside North America's `011`). `+41 79 960 50 92` and `0041 79 960 50 92` route to the same physical phone. The "Swiss trunk-prefix" framing in the docstring is incorrect — the **national** trunk prefix in Switzerland is `0` (`079 960 50 92`), not `00`. `00` is universally the international-dial prefix.

**Failure mode.** If Director ever sends from:
- A handset configured with `00` international dialing (common on European GSM/landline)
- A WhatsApp port (Web, Desktop, third-party API) that preserves the `00` prefix instead of stripping it
- A meeting transcript organizer field where the calendaring app rendered the number with `00`

…the C2 Inv 5 author-authority short-circuit fails silently. The Director's message proceeds to rule walk, may match `wa_minimum_content_length` (a curt "ok" reply) or be hashed as a duplicate, and gets dropped. Director's own content is the canonical never-drop case; this is the exact failure C2 was added to prevent.

**Fix.** One of:

- **(a)** Strip leading `00` after digit extraction:
  ```python
  digits = re.sub(r"\D", "", raw)
  return digits[2:] if digits.startswith("00") else digits
  ```
  Single line; canonicalizes both `+41…` and `0041…` to the same E.164 form. Trivial.

- **(b)** Pre-canonicalize known `00` variants in `DIRECTOR_PHONES`:
  ```python
  DIRECTOR_PHONES: frozenset[str] = frozenset({"41799605092", "0041799605092"})
  ```
  Two lines; works but doesn't generalize if Director's number rotates or another phone is added.

Lean (a). It fixes the class of bug, not just the instance, and matches how E.164 normalization is canonically done.

**Test.** Add one parametrize case to `test_normalize_phone_strips_non_digits`:
```python
("0041 79 960 50 92", "41799605092"),
("00 41 79 960 50 92", "41799605092"),
```

## 4. Nice-to-have

### N1 — `_process_layer0` raises mid-pipeline after PASS decision is made

**Location:** [`kbl/layer0.py:436-444`](../../kbl/layer0.py).

```python
if decision.verdict == "pass":
    if conn is not None and signal.raw_content:
        h = layer0_dedupe.content_hash(signal.raw_content)
        layer0_dedupe.insert_hash(conn, ...)  # raises on DB blip
    return decision  # never reached on raise
```

If `insert_hash` raises (DB blip mid-tick), the decision is `pass` but the function never returns it — caller sees the exception, not the PASS verdict. The caller must reconstruct: "exception means the decision was potentially complete but a side-effect failed."

**Two acceptable shapes:**
- **(a)** Catch + log + return PASS — forgive transient dedupe-write failures, signal proceeds without being deduped (next copy may also pass; acceptable trade for liveness)
- **(b)** Document explicitly in the function docstring that side-effect failures bubble after the decision is finalized, and pipeline_tick must treat the exception as "evaluate again from scratch" (relies on `ON CONFLICT DO NOTHING` for retry idempotency — which is in place)

Currently neither (a) nor (b). The pipeline_tick caller is a separate ticket; pin the contract before that ticket is written so behavior isn't accidentally divergent. Lean (b) — the rollback-and-raise pattern is consistent with the dedupe module's established posture.

### N2 — Same shape for `kbl_layer0_review_insert` on DROP path

**Location:** [`kbl/layer0.py:447-459`](../../kbl/layer0.py).

If review-insert raises during the 1-in-50 sample, the DROP decision was already made but the audit row is lost AND the function raises. Caller doesn't get the decision. Same trade-off as N1 — likely (b) document, but pin it.

### N3 — Review sampling is time-correlated (`signal.id % 50`)

**Location:** [`kbl/layer0.py:404, 448-451`](../../kbl/layer0.py).

`signal.id % 50 == 0` fires deterministically on signals 50, 100, 150, … BIGSERIAL allocates IDs in arrival order, so the sampled set is **time-correlated** — one busy ingest hour (e.g., a newsletter-sender blast at 09:00 dropping 30 signals in 5 min) might produce the only sample-hit for that day, biasing the audit set toward that hour's source mix.

For Phase 1 (~50 signals/day, 1 sample/day in expectation), the bias is small and the determinism wins. At Phase 2 scale (1000+ signals/day, 20 samples/day) the bias compounds — Director's audit view of "which Layer 0 drops were correct?" could over-represent newsletter-blast hours and under-represent low-volume sources (meetings, WA).

**Defer-but-track.** B3 spec'd modulo-id deliberately for reproducibility in tests; agree for v1. Re-evaluate at Phase 2 design — options: (a) per-source modulo (50 samples/source), (b) hash-of-content modulo (content-determined; still reproducible per-fixture), (c) reservoir sampling (uniform over the day; less reproducible).

Not blocking. Worth a note in §3.5 of the Step 0 spec for the Phase 2 review.

### N4 — Whole-word matcher doesn't tolerate multi-whitespace in multi-token aliases

**Location:** [`kbl/layer0.py:298-325`](../../kbl/layer0.py).

`_mentions_active_slug_or_alias` builds `re.search(rf"\b{re.escape(term.lower())}\b", body)`. Aliases are stored normalized by `slug_registry._normalize_key` (lowercase + single-space-collapsed). The **body** is not pre-normalized — `body = (content or "").lower()` only.

Failure case: body contains `"Mandarin   Oriental Vienna"` (multiple spaces between tokens, e.g., from a poorly-formatted email or copy-paste artifact). The pattern `\bmandarin oriental\b` (single space, from re.escape on the normalized alias) won't match.

**Fix options:**
- **(a)** Pre-collapse whitespace in body: `body = re.sub(r"\s+", " ", body)`. One line. Plays nicely with the existing whole-word match.
- **(b)** Use `\s+` between tokens in the regex: `re.escape(term.lower()).replace(r"\ ", r"\s+")`. Slightly more complex; same effect.

Either is fine. (a) is clearer.

In practice email/transcript content rarely has runs of whitespace inside named entities — low-frequency edge case. Defer until observed.

### N5 — `_kbl_conn` smuggled through `signal.payload`

**Location:** [`kbl/layer0.py:430-434, 207`](../../kbl/layer0.py).

```python
signal.payload.setdefault("_kbl_conn", conn)
try:
    decision = evaluate(signal, ...)
finally:
    signal.payload.pop("_kbl_conn", None)
```

This mutates the caller's payload dict to thread the connection through to `_content_hash_seen_within_hours` without widening the handler signature. The `try/finally` cleans up, and `test_process_layer0_clears_conn_from_payload_after_evaluate` guards the post-condition. ✓

**Two cosmetic concerns:**
1. **Reentrancy.** If a future caller is already using `_kbl_conn` as a payload key for an unrelated reason, `setdefault` won't overwrite but `pop` will remove it — losing the caller's data. Probability: ~0 today; key starts with `_` to signal "private". Worth a docstring note: "Reserved payload key `_kbl_conn` — do not use externally."
2. **Cleaner alternative.** A `contextvars.ContextVar` for conn would be more pythonic and side-effect-free. But it's heavier ceremony than the current shape merits, and the smuggled-key pattern is defensible at this scale. Defer unless the handler signatures grow more variants.

Not a fix request. Just naming the contract.
