## 3. Should-fix

### S1 — `kbl_layer0_review.review_verdict` lacks CHECK constraint

**Location:** migration line ~74.

```sql
review_verdict  TEXT  -- 'correct_drop' | 'false_positive' | 'ambiguous', NULL if unreviewed
```

The vocabulary is documented in the comment but not enforced by a CHECK constraint. My original S6 spec (in `B2_step0_layer0_rules_review_20260418.md`) had:
```sql
CHECK (review_verdict IN ('correct_drop', 'false_positive', 'ambiguous') OR review_verdict IS NULL)
```

**Why this matters.** Without the CHECK, Director-facing UI bugs or typos write arbitrary strings (`"good"`, `"wrong"`, `"FALSE_POSITIVE"`) — silent data quality drift. The 3-value vocabulary is small and stable; CHECK is cheap.

**Fix.** Either add the CHECK constraint (preferred) or document explicitly in the migration header that vocabulary is application-enforced and provide a list-of-known-values reference for KBL-C UI authors. 1 line of SQL.

### N2 — `signal_excerpt TEXT NOT NULL` has implicit-500-char-truncation contract

```sql
signal_excerpt  TEXT NOT NULL,  -- first 500 chars of payload for quick Director scan
```

Truncation responsibility lives in the writer (KBL-B impl). If a writer forgets to slice, the full payload could land in `signal_excerpt`. Database accepts it. UI assumes 500 chars.

**Fix.** Either add a CHECK `(char_length(signal_excerpt) <= 500)`, or document in the writer-side spec. CHECK is defensive but rejects oversize writes (might be undesirable). Lean: document in writer spec, no SQL change.
