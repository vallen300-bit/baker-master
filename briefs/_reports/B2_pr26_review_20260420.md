## Decisions on B3's flagged deviations

### Deviation A — PR #24 ranking (accept)

**Brief expected:** `#37 + #39` in top-5 for PR #24 (FEEDLY_WHOOP_KILL).

**v2 produces:** `#26 / #33 / #24 / #34 / #31` in top-5. `#37` + `#39` absent.

**Decision: ACCEPT v2's ranking.** Semantic analysis:

| v2 Rank | Lesson | Relevance to PR #24 (dead-sensor retirement) |
|---|---|---|
| #26 | "Wrong import path = feature silently dead since day one" | **Direct semantic match** — PR #24 retires silently-dead sensors. Best lesson for this PR. |
| #33 | "Vault structure: optimize for machine retrieval" | Tangential; token-driven (agent/skills/memory vocabulary in `.claude/agents/`). Borderline FP. |
| #24 | "Run periodic health audits — silent failures accumulate" | **Direct semantic match** — dead sensors accumulated because no periodic audit caught them. |
| #34 | "Structural verification ≠ integration verification" | Applies to any dead-code removal PR. |
| #31 | "SSE MCP transport — session ID mismatch on reconnect" | False positive. |

**v1's ranking (`#37 / #39` in top-5):** #37 was about DDL-in-Python-`_ensure_*` which is adjacent because PR #24 deleted `_ensure_whoop_tables`, but the lesson's core pattern (DDL never runs on Render) wasn't the driving concern of PR #24 (deletion, not DDL). #39 was about Render env-var PUTs not auto-restarting — a post-merge Tier B item I flagged in my PR #24 verdict, NOT a concern of the PR itself.

**v2's ranking is strictly more faithful to the PR's intent.** Rejecting Deviation A would require removing IDF + `+`-only filter — defeating the whole point of the upgrade. The brief's `#37+#39` expectation was a frozen snapshot of v1's noisier output; it should not be the acceptance criterion for v2.

### Deviation B — heuristic swap (accept)

**Brief specified:** fallback iff `highest score < 2× lowest score (of top-5)`.

**B3 implemented:** fallback iff `positive_hits ≥ 80% of total lessons`.

**Decision: ACCEPT B3's rule.** Quantitative analysis of smoke cases:

| PR | Top-5 score ratio (high/low) | Positive hits / total | Brief's rule fires? | B3's rule fires? | Ground truth (should fire?) |
|---|---|---|---|---|---|
| #21 | 3.11 / 3.11 = 1.00× | 2 / 45 = 4.4% | ✅ YES (falsely) | ❌ no | ❌ no (strong signal — 2 lessons directly apt) |
| #22 | 29.18 / 15.53 = **1.88×** | 28 / 45 = 62% | ✅ YES (falsely) | ❌ no | ❌ no (strong signal — #36 + #37 + #42 all on point) |
| #24 | 7.61 / 3.81 = 2.00× | 19 / 45 = 42% | ❌ no | ❌ no | ❌ no (mixed — accept v2 ranking per Deviation A) |
| bv#3 | n/a (empty RANKED) | 0 / 45 = 0% | n/a | ✅ YES (via empty-RANKED) | ✅ YES (scaffold, no real signal) |

**Brief's rule would FALSELY fallback on PR #21 and PR #22** — both strong-signal PRs. That's the exact failure mode we're trying to avoid (hiding real lesson signal). B3's coverage-based rule correctly:

- Fires for scaffold PRs (bv#3 at 0% coverage, via the empty-RANKED cascade).
- Does NOT fire for strong-signal PRs regardless of compressed score ratios.

The 80% threshold is an empirical separator — validated against 4 real PRs spanning scaffold / dead-code / env-drift / alias-rename. Accept.

Today there is no such tooling — B2 pastes the raw output into a fenced code block inside `B2_verdict_template.md`. If a future CI check or dashboard ever parses this, `score (N.NN)` pattern is what they should match against. Worth one line in helper's header comment.

Suggest: `"lessons #<LATEST-8>-<LATEST>"` where LATEST is computed from `grep -c "^### "` of the lessons file. Small; can ship in a follow-up.
