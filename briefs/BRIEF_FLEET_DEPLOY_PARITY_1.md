**The one-line problem:** three ARM launchd jobs (`arm-alarm`, `arm-cadence`, `arm-semantic`) each self-install to a deploy dir and each expose a `--check` drift subcommand — but **`--check` proves the deployed worker *parses*, never that it *matches the repo source it was cut from*** (`install_arm_alarm_job.sh:55-58` literally documents this: *"Health = installed + loaded + worker parses"*). So a worker merged in the repo but never re-installed on the host — the exact "shipped but not deployed" gap that stalled item-A — reads **CLEAN**. And when a job *does* fail at runtime (DB unreachable, a source going non-200), the tolerant exit-0 design degrades it to a **silent AMBER** instead of a loud RED, and the alarm that finally fires **drops the specific failure names**. This brief closes all four.

## Problem (four findings, one theme: a host-side job can diverge from what we shipped, and fail silently when it does)

1. **`--check` has no repo↔deployed parity assertion (finding-1 root; item-A generalization).** `WORKER_SRC="${SCRIPT_DIR}/arm_alarm_check.sh"` (`install_arm_alarm_job.sh:33`) is the repo source; `WORKER_DEPLOY="${DEPLOY_DIR}/arm_alarm_check.sh"` (`:37`) is what launchd actually runs. `--check` (`:60-88`) verifies `WORKER_DEPLOY` exists, is executable, is loaded, and *parses* — but never compares it to `WORKER_SRC`. A merged-but-not-reinstalled worker (repo ahead of host) is invisible. This is the launchd-fleet form of "env PUT didn't deploy" — the item-A stall class.
2. **No fleet-level view of deploy state.** Each installer answers "am *I* drifted?" in isolation; there is no single artifact that says "these N host-side jobs should be at these N repo shas on this host," so no one can run a **one-shot fleet parity sweep** or diff the deployed fleet against a known-good manifest. "Is the whole host-side fleet current?" is un-answerable today.
3. **A runtime failure degrades to a silent AMBER instead of a loud RED (finding-1 db_unreachable + finding-2 cadence DEGRADED).** `arm_cadence_poll.sh` tolerates any transient failure with exit-0 (`:29,33`) and only *logs* `"DEGRADED snapshot … (a source returned non-200/unparseable)"` (`:146`); when the underlying DB is unreachable the poller writes no fresh snapshot, the marker goes stale, and the missing-marker policy reads it **AMBER** (`arm_alarm_check.sh:83-84`) — a DB outage is indistinguishable from install-day quiet. A cadence that is *actively degraded* must produce a RED verdict, not a log line.
4. **The alarm drops the specific failure names (finding-8), and MISSING_IS_RED can't be safely armed (finding-4).** When the alarm does fire, the escalation should name *which* source(s) failed (`report:stale`, `canary:ok=false`, `db_unreachable`) — the verdict tuples already carry the key (`arm_alarm_check.sh:151` "`<key>\t<incident-type|detail>`") but the human-facing alarm must surface them, not a generic "a source failed." And `ARM_ALARM_MISSING_IS_RED=1` — the fail-loud posture where a never-seen marker is RED — can't be turned on today without false-paging, because we have no proof a producer is actually deployed. Parity (findings 1+2) is the missing proof that unblocks the flip.

## Fix (four pieces; each EXTENDS a shipped surface — no forks)

### F1 — Repo↔deployed hash-parity in every installer `--check` (findings 1 + 2)
In each of the three installers' `--check` block, add a parity assertion: `sha256` of `WORKER_DEPLOY` must equal `sha256` of `WORKER_SRC`; on mismatch emit `[FAIL] deployed worker drifted from repo source (deployed <sha8> != repo <sha8> — re-run install)` and set `rc=1` → `RESULT: DRIFT`. Use the same portable hash the repo already uses elsewhere (prefer `shasum -a 256`, fall back to `sha256sum`; a helper `_sha256()` in a sourced `scripts/lib/parity.sh` keeps the three installers identical — DRY, single definition). **The plist is parity-checked too:** the *rendered* installed plist encodes `StartInterval` + the worker path; assert the installed `StartInterval` still equals the value the current installer would render (the interval-clamp check at `:74-85` already reads it — extend to compare, don't add a second reader). Parity is **additive to** the existing presence/loaded/syntax/interval checks; a CLEAN result now means *deployed == repo*, not just *deployed parses*.

### F2 — Fleet deploy manifest + one-shot parity sweep (finding-2, fleet view)
Add `scripts/arm_fleet_manifest.json` (checked into the repo) enumerating each host-side job: `{ job_label, installer, worker_src (repo-relative path), deploy_dir_env, expected_interval_s, host_role }`. Add `scripts/arm_fleet_parity.sh` — a **read-only, zero-side-effect** wrapper that, for every manifest entry, invokes that installer's `--check` and aggregates: prints one line per job (`CLEAN` / `DRIFT <reason>` / `NOT-INSTALLED`), exits non-zero if **any** job is drifted or missing, and — critically — a job listed in the manifest whose installer/worker is **absent on the host is RED, never skipped** (`MISSING_IS_RED` applied at the fleet layer: a job we expect and don't find is a failure, not silence). The manifest is the single source of truth for "what the host-side fleet should be"; the sweep is the machine answer to "is it?". The sibling `arm_*_drift_check.sh` sentinels gain a call to `arm_fleet_parity.sh` (or reuse its `_sha256`) so parity drift surfaces off-band on the same cadence as install-health drift — do NOT add a new launchd job for the sweep; it rides the existing sentinels + is runnable by hand and in the drill.

### F3 — DB-unreachable / degraded → RED verdict carrying failure names (findings 1 + 2 + 8)
Make an *active* degradation loud, distinct from install-day quiet:
- **`arm_cadence_poll.sh`:** when a source is unreachable specifically because the **DB/endpoint connection failed** (vs a parseable non-200), tag the snapshot with a machine field `"health": "db_unreachable"` (not just the generic DEGRADED log). Keep exit-0 (launchd crash-only tolerance is deliberate — `:29`), but the *snapshot content* now carries the RED-worthy reason.
- **`arm_alarm_check.sh`:** a source whose latest snapshot reads `health=db_unreachable`, or a cadence marker that is stale **after having once been fresh** (already RED per `:42-43`), yields a RED verdict — and the emitted alarm body/subject **lists the failing source keys** from the verdict tuples (`:151`), e.g. `RED: db_unreachable(cadence), report:stale` — never a nameless "a source failed." This is finding-8: the alarm carries *what* broke.
- Preserve the E15 alarm-fatigue guard: one incident per `source-key + incident-type` (`:47`), recovery + re-arm unchanged. Adding names to an existing incident's body is not a new incident.

### F4 — MISSING_IS_RED flip folded into the arming ladder, gated on a passing two-sided drill (finding-4)
`ARM_ALARM_MISSING_IS_RED=1` is the fail-loud end state (a never-seen marker = RED). It ships **OFF** and is armed only at the end of a stepped ladder, each step an explicit lead GO, exactly mirroring the BTUNE arming-ladder pattern (`BRIEF_CASE_ONE_BTUNE_STARTED_SLO_TERMINAL_1.md` §Rollout). The **precondition for the flip is a passing two-sided drill** (below): parity must prove every producer marker's job is deployed + current on the host, so "missing = RED" cannot false-page a producer that was simply never installed.

### The two-sided drill set = the written GO-gate per ladder step (the deliverable that makes each arming step provable)
"Two-sided" = every drill exercises **both** the repo/installer side **and** the live host side, so a green drill means the code is right *and* it's actually running:
- **Repo side (hermetic, in `scripts/tests/`):** extend `test_arm_alarm.sh` / `test_arm_cadence.sh` (senders/launchctl injected as recorders, per the existing suites) — (a) parity PASS: deploy the current worker, `--check` → CLEAN; (b) parity FAIL: mutate the deployed worker one byte, `--check` → DRIFT with the sha-mismatch `[FAIL]`; (c) fleet sweep: a manifest with one entry not-installed → `arm_fleet_parity.sh` exits non-zero + prints RED for that job; (d) db_unreachable snapshot → cadence marks `health=db_unreachable` → alarm verdict RED naming `db_unreachable`; (e) MISSING_IS_RED=1 with a never-seen marker → RED (proving the flip does what the ladder will arm).
- **Host side (live, in the drill runbook, run by deputy as bus-health owner):** on the actual ARM host — install → `arm_fleet_parity.sh` CLEAN; then deliberately stale one deployed worker (edit the deployed copy) → sweep detects DRIFT within one sentinel cycle; re-install → back to CLEAN. This is the **written GO-gate**: no arming-ladder step advances until its two-sided drill (repo test green **and** the live half demonstrated) is recorded.
